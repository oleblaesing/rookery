# ADR-0024 — DNS, TLS, and WKD: advanced-method WKD, ACME HTTP-01, MTA-STS, DKIM key strength

**Status:** Accepted  
**Date:** 2026-05-23  
**Deciders:** project founders

---

## Context

Running a mail server requires co-ordinating several distinct DNS- and TLS-dependent
protocols:

- **WKD** (Web Key Directory) — serves users' public PGP keys over HTTPS so external
  clients can discover them without a keyserver lookup.
- **ACME / Let's Encrypt** — provisions and auto-renews TLS certificates for the web UI
  and for every HTTPS-dependent protocol (WKD, MTA-STS).
- **MTA-STS** (RFC 8461) — publishes a policy over HTTPS that tells sending servers to
  enforce TLS when delivering to this domain, and what to do if TLS fails.
- **DKIM** — signs outbound mail so receiving servers can verify the message was sent by
  an authorised sender for the From domain.

Each protocol has multiple implementation options. This ADR records the choices and
their rationale.

---

## Decision

### WKD method: advanced method only

rookery implements the **WKD advanced method** exclusively:

```
https://openpgpkey.<domain>/.well-known/openpgpkey/<domain>/hu/<z-base-32-sha1>
```

The **direct method** (`https://<domain>/.well-known/openpgpkey/...`) is not
implemented.

**Rationale:** The direct method serves WKD paths from the apex domain. Users who run
their own website at `alice.com` cannot point their apex at rookery's WKD endpoint
without breaking their site. The advanced method requires only a CNAME from
`openpgpkey.<domain>` to our infrastructure, leaving the apex domain free. This is
essential for custom-domain users (Phase 4 and §5.2b of PLAN.md).

**Implication:** the server must provision a TLS certificate for
`openpgpkey.<domain>` for every domain it manages, including the primary domain. This
is handled by ACME HTTP-01 below.

### ACME challenge type: HTTP-01

rookery uses **ACME HTTP-01** for all certificate provisioning:
- The primary domain (`<domain>`) and `openpgpkey.<domain>` — provisioned directly.
- Per-user custom domains: `openpgpkey.<user-domain>` and `mta-sts.<user-domain>` are
  pointed at our infrastructure via CNAME; HTTP-01 challenges for these hostnames are
  served by our server and Let's Encrypt follows the CNAME.

**DNS-01 is not required.** Requiring operators to integrate with a registrar API for
DNS-01 would add significant friction to the 30-minute-to-instance NFR (§10 of
PLAN.md). HTTP-01 with CNAMEs covers all use cases without registrar API access.

**Implementation:** `certmagic` handles ACME account registration, challenge serving,
certificate storage, and auto-renewal transparently. The ACME account is persisted in
Postgres (encrypted with the server master key, §11.6 of PLAN.md). `certmagic` is the
one new Go dependency this phase introduces.

**Certificate storage:** certificates are stored alongside ACME account data in Postgres,
encrypted at rest with the server master key. They are never written to the container
filesystem, so container restarts do not trigger re-provisioning.

### MTA-STS policy mode: `enforce` after verification

MTA-STS is published for all managed domains:

- **Primary domain** (Phase 3): policy published at
  `https://mta-sts.<domain>/.well-known/mta-sts.txt`; ACME provisions the cert; DNS
  TXT record `_mta-sts.<domain>` contains the policy ID.
- **Custom domains** (Phase 4): same pattern, with CNAME from `mta-sts.<user-domain>`
  to our infrastructure.

**Policy progression:**
- `testing` mode for the first 48 hours after a domain is activated. This allows
  operators and users to catch DNS misconfiguration without breaking inbound mail.
- `enforce` mode after 48 hours. In `enforce` mode, well-behaved senders refuse to
  deliver if TLS negotiation fails, which protects our users from downgrade attacks.

**Per-domain kill switch:** each domain has an operator-accessible `mta_sts_mode`
override in the database. Setting it to `testing` (via `psql`) temporarily downgrades
the policy — for use during cert renewal incidents or DNS transitions. Default is
`enforce` after the initial 48-hour window.

**TLS-RPT:** a `_smtp._tls.<domain>` TXT record is published alongside the MTA-STS
record. Receiving servers send TLS failure reports to the `rua` address. Reports are
logged and surfaced in Phase 5 observability metrics; there is no in-app dashboard in v1.

**Cert expiry alert threshold:** the server logs a warning ≥14 days before any managed
cert expires. This threshold is the same for primary-domain and custom-domain certs.
Cert expiry breaking MTA-STS (inbound mail silently dropped by senders in enforce mode)
is the worst failure mode — the 14-day window gives operators time to act. This becomes
a Prometheus metric in Phase 5.

### DKIM key strength and selectors

Each domain gets **two DKIM keypairs**:

| Selector | Algorithm | Rationale |
|---|---|---|
| `rookery-ed25519` | Ed25519 (RFC 8463) | Modern, compact, fast. Primary signature. |
| `rookery-rsa` | RSA-2048 | Fallback for receivers that don't support ed25519 — still common (some legacy MTAs, some spam filters). |

Both selectors are published in DNS. Outbound mail carries both signatures; receiving
servers use whichever they understand.

**Key generation:** per-domain DKIM keypairs are generated server-side at domain
verification time. Private keys are encrypted at rest with the server master key
(ADR-0023 / §11.6 of PLAN.md). Public keys are published via DNS TXT records (primary
domain) or via CNAME to instance-level selectors (custom domains).

**Key rotation:** stub only in Phase 3 (primary domain) and Phase 4 (custom domains).
Full dual-selector rotation tooling (publish new key, wait for TTL, retire old key,
alert operator) lands in Phase 8.

## Consequences

**Positive:**
- WKD advanced method allows custom-domain users to keep their apex domain free;
  certmagic handles multi-domain certs transparently via HTTP-01.
- No registrar API integration required; operators only need to publish CNAMEs.
- MTA-STS `enforce` mode gives our users strong inbound TLS guarantees once the
  initial testing window passes.
- Dual DKIM selectors (ed25519 + RSA-2048) maximise compatibility without
  weakening the modern signature.
- certmagic's automatic renewal removes the most common operational failure mode
  of self-hosted TLS (forgotten cert renewal).

**Negative / trade-offs:**
- WKD advanced method requires a cert for `openpgpkey.<domain>` — one extra
  certificate to provision and renew per domain. This is handled automatically by
  certmagic; the operational cost is negligible.
- MTA-STS `testing` → `enforce` progression adds a 48-hour window before full
  protection kicks in. This is the right trade-off: `enforce` from day one would
  break inbound mail for any operator whose DNS is misconfigured.
- ACME HTTP-01 requires port 80 to be reachable from the internet (for Let's
  Encrypt's challenge servers). This is standard; operators with firewalls that block
  port 80 need to open it. Documented in the operator runbook.
- certmagic is a new Go dependency. Justified: implementing ACME from the stdlib
  `golang.org/x/crypto/acme` client is several hundred lines of error-prone protocol
  handling that certmagic has already done correctly and tested in production.

## Alternatives considered

- **WKD direct method as a fallback:** rejected. Supporting both methods doubles the
  serving surface and the ACME provisioning paths for no benefit — the advanced method
  is supported by all modern WKD clients.
- **DNS-01 challenges:** rejected for the primary reason above (registrar API
  dependency). Reconsidered only if an operator genuinely cannot use port 80, which is
  an extremely rare edge case.
- **`golang.org/x/crypto/acme` directly instead of certmagic:** possible, but certmagic
  adds certificate storage, renewal scheduling, and multi-domain handling on top of the
  bare ACME client. Replacing it would require several hundred lines of infrastructure
  that we would then own and maintain. Not justified under P12.
- **Self-signed or operator-provided certificates:** a configuration option for operators
  who want to bring their own TLS (e.g. from their load balancer or CDN). Not in Phase 3
  scope; could be added later as a `tls_mode = "external"` config flag if there is
  demand. The security implications (operator-managed renewal, potential misuse) are
  documented in that ADR if it is ever written.
- **MTA-STS `enforce` from day one:** rejected. Operators who make a DNS mistake during
  initial setup would silently lose inbound mail. The `testing` window is a safety net.
