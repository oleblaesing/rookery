# ADR-0002 — Server-rendered HTML, hand-written JS only (crypto + partials); no SPA, no HTMX

**Status:** Accepted  
**Date:** 2026-05-15  
**Deciders:** project founders

---

## Context

The rookery web client must support PGP operations (key generation, encrypt, decrypt,
sign, verify) in the browser. This is a hard constraint: the private key never leaves
the user's device, so the server cannot perform these operations on the user's behalf.
Some JavaScript is therefore unavoidable.

The question is how much JavaScript, of what kind, and organised how.

## Decision

The web UI is **server-rendered HTML** using Go's `html/template` package. JavaScript
is restricted to two tightly-scoped roles:

1. **Crypto module** (`web/crypto/`): handles all PGP and Argon2id operations on the
   compose and read pages. Uses `OpenPGP.js` (pinned, SRI-locked) and a WASM Argon2id
   binary (pinned, SRI-locked). Bundled by `esbuild` inside the Containerfile build.
   The output is a single `crypto.js` file under 200 KB unminified (§10 of PLAN.md).

2. **Partials module** (`web/partials/partials.js`): hand-written, no third-party
   library, no build step. Provides `fetch + swap` helpers for the handful of partial-
   page update needs: recipient key-status hints, DNS verification polling on the
   custom-domain setup page, mark-as-read. Endpoints return **HTML fragments**, not
   JSON, except where the crypto module needs raw data. See ADR-0012 for the full
   rationale and contract.

There is **no SPA** (no React, Vue, Svelte, Solid, etc.), **no HTMX** (see ADR-0012),
and **no client-side router**. Pages are full HTML documents served by Go handlers.

Third-party scripts are **never loaded from external origins**. The only third-party
browser dependencies accepted are `OpenPGP.js` and the WASM Argon2id binary, and these
are vendored or pinned with SRI hashes committed to the repo. See ADR-0013.

## Consequences

**Positive:**
- The JS surface that touches keys and plaintext is small and auditable in an afternoon.
- A strict Content Security Policy (no `unsafe-inline`, no third-party origins) is
  achievable without complex build-time tooling. See ADR-0016.
- Server-rendered pages work correctly when JS is disabled (with the expected
  degradation: compose and read pages show "JS required for PGP"; other pages are
  fully functional).
- No client-side routing means no need for a history API, no "blank page on reload"
  problems, no SEO plumbing.
- `html/template` is a Go stdlib package — zero dependencies, known behaviour.
- Partial updates (`partials.js`) are handled without a library dependency because the
  use cases fit in ~200 lines of hand-written code (ADR-0012).

**Negative / trade-offs:**
- Every navigation is a full page load. For the target audience (people who use
  terminal applications and know what a keypair is) this is entirely acceptable.
- Implementing real-time inbox updates via SSE (Phase 4+) requires some JS, which
  `partials.js` handles via `EventSource` — a browser stdlib API.
- The compose experience is not as smooth as a full SPA. This is a deliberate match
  to the v1 scope (§8 Phase 1-2 of PLAN.md): utilitarian, not Superhuman.

## Alternatives considered

- **React / Vue / Svelte SPA:** rejected. Supply-chain risk (npm cascades), large JS
  bundle, no meaningful improvement to functionality given v1's scope. Contradicts P11
  and P12 of PLAN.md.
- **HTMX:** rejected in ADR-0012. Supply-chain reasoning and the partial-update needs
  being small enough to own are the main reasons.
- **Inertia.js or similar hybrid:** rejected. Same supply-chain concerns, more
  complexity than the use case justifies.
- **No JS at all:** rejected. PGP operations must happen in the browser; the private
  key cannot go to the server (ADR-0010). A zero-JS UI would require a server-side
  key model, which is the architecture we explicitly rejected.
