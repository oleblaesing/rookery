# ADR-0017 — Address model: multiple addresses per user, plus-addressing, aliases, catch-all

**Status:** Accepted
**Date:** 2026-05-23
**Deciders:** project founders

---

## Context

A single user account is associated with a PGP keypair (ADR-0014: key-as-identity,
address-as-attribute). Addresses are mutable attributes of that identity. Phase 4 makes
this concrete: a user can hold addresses on the primary domain, on one or many verified
custom domains, and can configure aliases and catch-alls on domains they own.

This ADR records the address model decisions (from §11.3 of PLAN.md) as a formal record
now that Phase 4 implementation is beginning.

---

## Decision

### Multiple addresses per user account

A single user account and its associated PGP key can hold one or many addresses. Example:
Alice holds `alice@personal.example`, `alice@work.example`, and `a@a.li` — all bound to
the same inbox, the same key, and the same session.

**Data model:** the existing `addresses` table is the canonical source. Each address row
carries `user_id`, `domain_id`, and `local_part`. The `users.primary_address_id` column
points to the address used as the default `From` on compose.

**Setting up additional addresses:** a user adds an address on any domain they own (verified
custom domain) or on the primary instance domain. The server validates domain ownership
before inserting the row. Reserved local-parts (ADR-0018) are blocked.

**Removing an address:** any non-primary address can be removed. Removing the primary address
is blocked; the user must first designate another address as primary.

### Default From address

`users.primary_address_id` is the default From. The compose page pre-selects it. The user
can override per-message by selecting any of their other addresses from a dropdown.

**Invariant:** `primary_address_id` is always non-null for a registered user. The first
address created at signup becomes the primary; subsequent addresses are added non-primary
by default.

**Changing primary:** user toggles "Set as default" in account settings (address list). The
server updates `users.primary_address_id`.

### Plus-addressing

`alice+tag@domain` routes to `alice@domain` for every address by default. The `+`-tag and
everything after it is stripped during recipient resolution in the inbound SMTP backend
(`smtp/inbound.go` `resolveRecipient`). The `addresses.plus_addressing_enabled` column
allows per-address opt-out (rarely needed; the default is enabled).

**No filter-on-tag in v1.** The tag reaches the inbox in the `to_addresses` header column
for user-side filtering (Phase 8 will surface tags as inbox filter criteria if demanded).

### Aliases

An alias is an address row with `is_alias = TRUE` and `alias_target_id` pointing to a
canonical address belonging to the same user. Example: `support@alice.com` → `alice@alice.com`.

**Aliases are per-user** (set up in account settings). Operator-level aliases at the domain
(e.g. `postmaster@` routing to the owner) are created via the same mechanism but inserted
programmatically at domain-verification time — see ADR-0018.

**Alias behaviour:** inbound mail to an alias is delivered to the target address's inbox
with the original recipient preserved in the envelope and in a `Delivered-To` header. No
bounce is generated for a valid alias even if the target user is over-quota (quota check is
on the canonical address).

**Alias resolution order** in `resolveRecipient`:
1. Strip plus-tag.
2. Look up exact match in `addresses`.
3. If `is_alias = TRUE`, resolve to `alias_target_id` (one hop — no chains in v1).
4. If still not found and catch-all is enabled on the domain, route to catch-all address.
5. If not found, reject with `550 5.1.1 No such user`.

### Catch-all on custom domains

Per domain, a user who owns that domain can enable catch-all (`domains.catch_all_enabled`).
When enabled, `*@<domain>` for any local-part that does not resolve to an existing address
is delivered to `domains.catch_all_address_id` (a specific address owned by the domain owner).

**Off by default** — catch-alls attract spam. The domain settings UI has an explicit opt-in
toggle. Enabling catch-all requires the domain to be verified and owned by the authenticated
user.

**Rate limit:** catch-all inbound is subject to the same per-user rate limits as normal
inbound (§11.4). No separate catch-all rate limit in v1.

**Inbox indicator:** messages delivered via catch-all carry a `delivery_method = 'catch_all'`
marker in the message metadata. The inbox row shows "via catch-all: `<original-address>`"
so the user can distinguish catch-all traffic from addressed traffic.

**Catch-all routing rule:** the catch-all fires only after normal address resolution (including
alias resolution) finds no match. A message to `postmaster@alice.com` goes to the reserved
address, not to catch-all.

---

## Consequences

**Positive:**
- Multiple addresses per account enables the core §5.2b migration story: Alice owns
  `alice@alice.com` regardless of which instance she uses.
- Plus-addressing is free — no schema change, just local-part stripping.
- Aliases cover the "I want `support@` to go to my inbox" case without a separate account.
- Catch-all on custom domains covers power-user cases (disposable addresses, catch-all
  inboxes for a small business domain).

**Negative / trade-offs:**
- Multiple addresses per account complicates the compose From selector UI (minor; a simple
  `<select>` element).
- Catch-all can silently deliver spam to the user if they forget to turn it off after initial
  testing. The default-off and explicit indicator mitigate this.
- Alias chains are not supported (alias → alias). One hop only, enforced at write time
  (attempting to alias to an alias is rejected with a clear error).

## Alternatives considered

- **One address per account:** rejected. It breaks the §5.2b migration story and the
  key-as-identity / address-as-attribute model (ADR-0014).
- **Round-robin catch-all to multiple admins:** rejected. Unnecessary complexity for v1;
  the one-user-per-catch-all model is sufficient and simpler.
- **Alias chains (alias → alias → canonical):** rejected. Circular aliases would be a bug
  surface; two hops are rarely needed and the one-hop constraint is easy to explain.
- **Per-user catch-all (on any domain the user has an address on):** rejected. Catch-all is
  a domain-level setting controlled by the domain owner, not any address holder on that domain.
