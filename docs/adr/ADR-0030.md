# ADR-0030 — Smarthost integration (outbound relay and relay-rookery)

**Status:** Accepted
**Date:** 2026-05-29
**Deciders:** project founders

---

## Context

A fresh VPS IP gets its outbound mail junked by Gmail/Outlook for weeks or months,
regardless of how correct the operator's DKIM/SPF/DMARC/MTA-STS are (PLAN.md §9.1).
This is *the* problem that kills self-hosted email, and it is not one rookery can
solve directly — IP reputation belongs to the provider's range, not to us.

The realistic mitigation is a **smarthost**: route outbound through a trusted
upstream SMTP submission endpoint that already has delivery reputation. PLAN.md
§11.10 committed the *shape* of this feature and reserved this ADR number for the
detailed decision. It is being implemented standalone, ahead of its nominal Phase 8
slot, the way export/import (ADR-0039) was pulled forward.

Two flavours of upstream are in scope under one mechanism:

1. **Commercial relay** — SES, Postmark, Mailgun. Pay a vendor for guaranteed
   placement.
2. **Relay rookery** — another rookery instance with a clean IP, doing the favor for
   an instance whose IP has no history. To rookery, both are identical: "a remote
   SMTP submission endpoint we trust to relay our already-signed mail."

The symmetric capability — an instance *acting as* the relay rookery for whitelisted
downstream operators — is the second half of the feature and requires an
authenticated SMTP submission listener that rookery does not currently have (end
users submit over HTTPS; the only SMTP listener is unauthenticated inbound MX on
port 25).

The core invariants this design must not break:

- **Sign-then-handoff.** rookery DKIM-signs every outbound message with the user's
  domain key *before* the smarthost sees it. The smarthost is opaque transport and
  cannot impersonate the user cryptographically — it can only relay what we signed.
- **Not a federation protocol.** No directory, no auto-discovery, no DHT, no novel
  wire protocol. Relay trust is one bilateral, out-of-band agreement at a time,
  exactly like every inter-administrator relay trust on the open internet (PLAN §2
  principle 2, §3).
- **The server never holds a user's PGP private key.** Unchanged — the smarthost
  path is below the crypto layer entirely; for PGP mail the relayed body is already
  ciphertext.

---

## Decision

### 1. One config block, one wire shape: `[smtp.smarthost]`

A single optional block configures both commercial relays and relay rookeries; only
the hostname and credential vary. The pre-existing dev-only `relay_host`/`relay_port`
fields are **folded into this block** — there is one outbound-relay mechanism, not
two. Dev (capturing outbound in mailpit) becomes a smarthost with `auth = false`,
`require_tls = false`.

```toml
[smtp.smarthost]
enabled     = false   # off by default → direct MX delivery
host        = "smtp.postmarkapp.com"
port        = 587     # 587 STARTTLS or 465 implicit TLS
username    = "..."
require_tls = true    # mandatory TLS for a credential-bearing endpoint
auth        = true
```

The password is `ROOKERY_SMTP_RELAY_PASSWORD` (env var, never the file) — already
reserved in PLAN §11.11.

**Rejected:** keeping `relay_host` as a separate dev mechanism. Two ways to configure
"send through another host" is one too many; the dev case is just a smarthost with
the security knobs off.

### 2. Outbound relay: authenticated, TLS-enforced

When `enabled`, the outbound queue routes through the smarthost instead of direct MX.
DKIM signing already happens before the relay/direct branch in `queue.go`, so the
sign-then-handoff invariant holds without change.

The relay delivery path differs from MX delivery in two deliberate ways:

- **TLS is mandatory, not opportunistic.** A smarthost session carries AUTH
  credentials; if `require_tls` and the server won't do STARTTLS (or 465 implicit
  TLS), the attempt **fails** rather than falling back to plaintext. (Direct MX
  delivery keeps its opportunistic-and-continue behavior — there is no credential to
  leak there and unencrypted is better than undelivered.)
- **AUTH (SASL PLAIN/LOGIN over TLS).** Via the existing `go-sasl` dependency.

When the smarthost is down, it is treated as any other delivery failure: the existing
retry backoff and bounce-after-max-age machinery applies. No separate
smarthost-down timeout policy in v1.

### 3. Relay rookery: authenticated submission listener, whitelist of clients

To *act as* a relay rookery, an operator opts in (`submission_enabled = true`) and
rookery starts a **new authenticated SMTP submission listener** on 465/587, separate
from the inbound MX listener:

- AUTH is offered **only over TLS**; an unauthenticated session can never relay
  (no open relay, ever).
- Credentials live in a new `relay_clients` table — the **whitelist**. Each row is a
  SASL username + a hash of the issued secret, an enabled flag, a per-client hourly
  rate cap, and a human label. This is the first server-side stored auth secret in
  rookery; it is acceptable because relay clients are remote *operators* speaking a
  commercial-relay-shaped protocol, not end users (whose auth remains PGP
  challenge/response with nothing stored). The secret is hashed, never stored
  plaintext, and compared in constant time.
- An authenticated relay session may send to **any** recipient — that is the function.
  Accepted mail is enqueued into the **existing `outbound_queue`**, tagged with the
  originating `relay_client_id`, and delivered by the same worker. The relay rookery
  does **not** re-sign DKIM: the downstream already signed, and opaque transport is
  the whole security argument for why relaying another instance's mail is safe.
- Per-client rate limits (reusing the existing window-count pattern) plus the
  per-instance limits (PLAN §11.4) apply to relayed traffic. Over-limit returns a
  4xx so the downstream's queue retries.

**Credential provisioning is out-of-band.** `rookery relay-client create` generates a
secret, inserts the (hashed) row, and prints the credential + a ready-to-paste
`[smtp.smarthost]` block once. `list` and `revoke` round out the lifecycle. There is
no endpoint that auto-vends relay credentials and no rendezvous service — the
downstream operator gets credentials the same way they would get a Postmark API key.

### 4. TLS certs reused from Caddy

The submission listener needs certs for the instance hostname. Rather than add a
second ACME client, rookery reads the cert/key PEMs Caddy already provisions, via a
read-only mount of the `caddy-data` volume
(`/data/caddy/certificates/<ca>/<host>/<host>.{crt,key}`). Cert hot-reload on ACME
renewal is out of scope for v1 — a restart picks up the renewed cert (renewals are
~60-day events).

**Rejected:** built-in `autocert`. It would duplicate ACME, fight Caddy over the
port-80 HTTP-01 challenge, and add a dependency for a problem Caddy already solves.
**Rejected:** operator-supplied static cert paths as the primary mechanism — pushes
renewal onto the operator; offered only as an escape hatch if needed later.

## Consequences

**Good:**

- The deliverability story PLAN §9.1 promised becomes real and configurable, for both
  paying-a-vendor and leaning-on-a-friend.
- Swapping a commercial relay for a relay rookery (or back) changes only host +
  credential; message authentication is unchanged because we sign first.
- The relay-rookery half reuses the existing queue, DKIM, and rate-limit machinery;
  the genuinely new surface is one authenticated listener and one credentials table.

**Costs / risks named honestly:**

- **Abuse exposure for the relay rookery.** It inherits its downstreams' outbound
  reputation; downstream spam dirties the relay's IP. Unlike a commercial smarthost,
  a hobbyist operator has no abuse team, KYC, or lawyers. Mitigations: only relay for
  operators you know, the same rate limits as local traffic, and credential
  revocation as the v1 remediation. The README and runbook say this out loud.
- **First stored auth secret.** `relay_clients` introduces a server-side secret,
  contrary to the end-user no-stored-secret posture. Scoped to operators, hashed,
  revocable — and it never touches user keys or sessions.
- **A second SMTP listener and SMTP-side TLS** is new operational surface (ports
  465/587, cert mount). Off by default; only relay-rookery operators take it on.

**Out of scope (v1):** per-user/per-domain smarthost selection; multiple smarthosts /
failover; smarthost-specific retry timeouts; cert hot-reload; relayed-mail content
scanning beyond existing rate limits; any auto-discovery or directory.

Phased; see `docs/smarthost-plan.md` for the step-by-step. **Phase A** (use a
