# ADR-0033 — Unified `rookery` shell-script dispatcher

**Status:** Accepted
**Date:** 2026-05-23
**Deciders:** project founders
**Supersedes in part:** ADR-0005 (the "separate admin CLI binary" rejected
alternative), ADR-0008 (the `./scripts/new-invite.sh` invocation form). The
shell-first operator principle in both ADRs is unchanged.

---

## Context

The operator currently uses three surfaces to drive a rookery instance:

1. `./run.sh` — a host-side wrapper around `docker compose` that generates `.env`
   on first run and the `Caddyfile` on first `--profile prod` run, then `exec`s
   into `docker compose` with the original arguments.
2. `docker compose exec -T postgres sh /scripts/new-invite.sh [days]` — runs
   `new-invite.sh` inside the postgres container because the rookery container
   is distroless (no shell, no `psql`).
3. `./scripts/send-test-mail.sh ...` — runs on the host, talks to the dev
   stack's port 25 via `curl`.

Three things are wrong with this:

- **Two entry points, two mental models.** Operators have to know which script
  to run from where, and that `new-invite` is a `docker compose exec` away
  while `send-test-mail` is a host script. The README has to teach both forms.
- **Direct `docker compose` calls leak everywhere.** Despite the existence of
  `run.sh`, the README, PLAN.md, CLAUDE.md, the systemd unit, ADR-0005, and
  the script headers all show `docker compose ...` forms — partly for clarity,
  partly because some flows (`exec`, `logs`) were never wrapped by `run.sh`.
  The result is that the documented surface is "use `run.sh`, except when you
  shouldn't," which is exactly the kind of split surface P10 warns against.
- **No bootstrap, no system-install, no update verbs.** `run.sh` generates
  `.env` and `Caddyfile` as a *side effect* of `up`. There is no first-class
  "set this instance up" verb, no first-class "promote this checkout into a
  system service" verb (the README hand-rolls the systemd install with `cp`
  and `systemctl daemon-reload`), and no first-class "pull and rebuild" verb
  (it's a documented incantation, not a command). Every one of these is a
  one-liner that the operator should not have to memorise.

## Decision

**A single executable, `rookery`, is the operator's entry point for every
instance operation.** It is a POSIX shell script at the repo root (and
mirrored into the container image at `/opt/rookery/rookery` for in-container
invocation when relevant). It dispatches to subcommands. It replaces
`run.sh`, `scripts/new-invite.sh`, and `scripts/send-test-mail.sh`. All
documented operator flows go through `rookery <subcommand>`; direct
`docker compose ...` invocations disappear from documentation and tooling
(an escape hatch — `rookery compose <args...>` — remains for ad-hoc use).

**Shell script, not a Go binary.** The transparency objection ADR-0005 raises
against a "separate admin CLI binary" still holds: a shell script can be read
end-to-end in five minutes, has no build step, and is auditable before it is
run. Consolidating the existing shell scripts into one dispatcher does not
introduce a binary; it removes two of three and merges them with `run.sh`. The
script stays under ~500 lines, POSIX-compatible (`/bin/sh`), and depends only
on tools that are already prerequisites (`docker` with the Compose v2 plugin,
`openssl`, `curl`).

### Privilege model: `init` is user-local, `install` is the sudo step

`init` writes only user-owned files into the checkout: `.env`, `rookery.toml`
(from the example), `Caddyfile`, and a staged `./rookery.service`. It never
needs `sudo` and is idempotent — safe to re-run after a `git pull` that adds
new generated files. There is no `--dev` / `--prod` flag on `init`; it
generates the full set every time, because all of those files are cheap and
inert until something uses them. Dev contributors who never call `install`
end up with an unused `Caddyfile` and `rookery.service` in their working
tree, which is harmless (both are gitignored).

`install` is the explicit privilege escalation: it copies the staged
`./rookery.service` into `/etc/systemd/system/rookery.service` and runs
`systemctl daemon-reload`. **The operator runs `install` exactly once per
host**, and it is the only command in the dispatcher that needs `sudo`. It
does not enable or start the unit — `systemctl enable --now rookery` is the
operator's deliberate "yes, autostart this on boot" decision.

This split has the effect the previous iteration of this ADR was reaching
for via a `--prod` flag on `init`: the *act of running* `install` is the
sole signal that this checkout is a production deployment. Everything before
that point is mode-agnostic.

### v1 subcommand surface

| Subcommand | What it does | Replaces |
|---|---|---|
| `rookery init [--domain X] [--email Y] [--name N] [--user U]` | Bootstrap from a clone. Generates `rookery.toml` if missing; the flags pre-fill `domain`, `contact_email`, and `instance_name` (any not given are prompted interactively unless `--no-prompt`). Generates `.env` with random secrets if absent. Generates `Caddyfile` from `rookery.toml` if absent. Stages `./rookery.service` with `User=` set from `--user` (default: `whoami`) if absent. Idempotent — safe to re-run; never overwrites a file that already exists. User-local only; never needs `sudo`. | the implicit `.env` + `Caddyfile` generation in `run.sh` |
| `sudo rookery install` | Copies `./rookery.service` to `/etc/systemd/system/rookery.service` and runs `systemctl daemon-reload`. Does **not** enable or start the unit. Run once per host. | the manual `sudo cp docs/ops/rookery.service ...` step in the README |
| `rookery start [--prod]` | Bring the stack up. Defaults to **dev** (no Caddy, mailpit on 1025/8025, web UI on 8080); `--prod` enables Caddy on 80/443 with TLS. The systemd unit always passes `--prod` explicitly; operators rarely type `start` directly. Errors if `rookery init` has not been run. | `./run.sh --profile dev/prod up --build` |
| `rookery stop` | Stop the stack (`docker compose down`). | `docker compose down` |
| `rookery restart [--prod]` | `stop` then `start`. Same `--prod` semantics as `start`. | `docker compose down && docker compose up` |
| `rookery update` | Upgrade in place: `git fetch && git pull --ff-only && docker compose build`. Does **not** restart — the operator runs `systemctl restart rookery` (prod) or `rookery restart` (dev). `--ff-only` makes a divergent local branch fail loudly instead of merging silently. | the documented `git pull && docker compose up -d --build` |
| `rookery logs [service]` | Tail logs from one service (default: rookery). | `docker compose logs -f rookery` |
| `rookery ps` | Show running services. | `docker compose ps` |
| `rookery invite create [validity_days]` | Generate an invite URL. Internally runs `psql -c "INSERT INTO invites ..."` inside the postgres container and `printf`s the URL. The noun-verb form (`invite create`) leaves room for `invite list` / `invite revoke` later without breaking the API. | `docker compose exec -T postgres sh /scripts/new-invite.sh [days]` |
| `rookery send-mail [flags] <to> [from] [subject] [body]` | Inject a message into the dev stack's inbound SMTP. Accepts the existing flags (`--encrypted`, `--pubkey`, `--fetch-key`). Named `send-mail` rather than `send-test` because it is the operator-facing way to inject a message and reads naturally in non-test contexts (smoke checks, manual reproduction). | `./scripts/send-test-mail.sh ...` |
| `rookery psql` | Drop into a `psql` shell inside the postgres container. | `docker compose exec postgres psql -U rookery -d rookery` |
| `rookery test` | Run the Go test suite. | `docker compose --profile test run --rm test` |
| `rookery vet` | Run `go vet`. | `docker compose --profile lint run --rm lint` |
| `rookery exec <service> <cmd...>` | Escape hatch: `docker compose exec` against any service. | `docker compose exec <service> <cmd...>` |
| `rookery compose <args...>` | Escape hatch: arbitrary `docker compose` invocation, with `.env` / `Caddyfile` generation skipped. | direct `docker compose ...` |
| `rookery help [subcommand]` | Usage. | — |

### Phases 5–6 will extend the surface

The operator-script set committed in PLAN.md §11.12 becomes additional
subcommands as their phases land. Where it makes sense, they follow the
noun-verb form (`<noun> <verb>`) introduced by `invite create`:

- Phase 5: `rookery user suspend <address>`, `rookery user unsuspend <address>`,
  `rookery user delete <address> [reason]`, `rookery dns print [domain]`,
  `rookery stats print`, `rookery master-key rotate`. (Or flat verbs —
  `suspend-user`, `print-dns` — if the noun grouping reads worse than the
  flat form. The dispatcher source decides per case.)
- Phase 6: `rookery backup`, `rookery restore <path>` (ADR-0029).
- Phase 8: `rookery relay-client create` if and when the relay-rookery shape
  lands (ADR-0030).

Each of these is a few dozen lines of shell inside the same dispatcher —
typically a small `psql -c "..."` plus a `printf`. The dispatcher source remains
the canonical reference; no separate per-task script files.

### Headline flows

**Operator, fresh VPS:** clone into `/opt/rookery` so the systemd unit's
`WorkingDirectory=/opt/rookery` lines up without editing.

```sh
sudo mkdir -p /opt/rookery && sudo chown "$USER" /opt/rookery
git clone <url> /opt/rookery && cd /opt/rookery
./rookery init --domain rookery.example --email you@example.com --name "My Rookery"
sudo ./rookery install
sudo systemctl enable --now rookery
# publish DNS records from `./rookery logs | grep DNS`
./rookery invite create
```

**Operator, upgrade:**

```sh
cd /opt/rookery
./rookery update
sudo systemctl restart rookery
```

**Dev contributor, fresh clone:** clone wherever — the dev path doesn't go
through systemd, so the `/opt/rookery` convention is prod-only.

```sh
git clone <url> rookery && cd rookery
./rookery init                # accepts the example defaults; can be edited later
./rookery start               # dev stack on :8080 with mailpit on :8025
./rookery test                # in another terminal
```

### What does *not* change

- The operator's interface is still **the shell**. There is no admin web UI, no
  in-app admin role, no setup wizard. ADR-0005's shell-first principle is
  intact.
- The single in-app role is still **user**. ADR-0008's "one role" decision is
  intact. The first-user bootstrap is still `rookery invite create` → visit
  the URL, same flow as every subsequent user.
- The data-model constraint from PLAN.md §11.12 still binds: every
  operator-meaningful state change must be a small invariant-preserving row
  update. The dispatcher is a thin wrapper around `psql`, not a place for
  business logic. If a future subcommand grows beyond ~30 lines of shell, that
  is a signal the schema needs flattening or the work belongs in the Go server,
  not that the dispatcher needs a framework.
- `compose.yaml` and `rookery.toml` are unchanged in structure. The dispatcher
  reads them; it does not own them.

## Consequences

**Positive:**

- One entry point, one mental model. The README's operator quickstart is four
  commands (`init`, `install`, `systemctl enable --now`, `invite create`) and
  the operator never types `docker compose` directly.
- The privilege boundary is explicit: `init` is user-local, `install` is the
  one sudo step, `start` and friends are user-local again. The README does not
  have to thread a `sudo` through a single command.
- `init` is mode-agnostic, which means `init --prod` and the surrounding
  branching go away. There is exactly one way to bootstrap; promotion to a
  system service is a separate verb.
- `update` and `install` give the upgrade path and the systemd install path
  the first-class verbs they currently lack. Both were previously documented
  incantations the operator had to remember.
- The dispatcher is still a shell script, so the ADR-0005 transparency
  argument (`cat rookery` prints what runs) holds.
- The escape hatch (`rookery compose ...`) means operators who need an
  un-wrapped `docker compose` call still have one, without us having to
  pretend the wrapper is exhaustive.
- The noun-verb form (`invite create`, future `user suspend`) leaves clean
  room for additional verbs without breaking the API. `invite` alone would
  have to grow flags or become an alias for `invite create` later.

**Negative / trade-offs:**

- The dispatcher is now ~500 lines of shell instead of 100. Larger, but still
  inspectable in a single sitting, and the alternative (a Go binary with its
  own argument parser, help text generator, and test suite) is much larger.
- Subcommand-style CLIs invite scope creep. Mitigation: every new subcommand
  must justify why it cannot be a documented `psql` query or a
  `rookery exec ...` one-liner. The data-model constraint above is the real
  brake.
- `init`'s interactive prompts are convenient for the README flow but
  inconvenient for Ansible / Terraform-style automation. Mitigation: flags
  (`--domain`, `--email`, `--name`, `--user`, `--no-prompt`) cover the
  non-interactive path; the README documents both.
- Dev contributors who never run `install` end up with an unused `Caddyfile`
  and `rookery.service` in their working tree. Cost: two extra files,
  gitignored. Worth the simpler `init` surface.
- Two existing scripts (`new-invite.sh`, `send-test-mail.sh`) and `run.sh`
  are deleted from the repo. The logic moves into the dispatcher. The
  compose-file bind-mount (`./scripts:/scripts:ro` on the postgres service)
  is removed; the dispatcher invokes `psql` directly via `docker compose exec
  postgres psql ...` instead of relying on a script inside the container.

## Alternatives considered

- **Keep run.sh + per-task scripts, add a `rookery-cli` Go binary on top.**
  Rejected: introduces a binary where one is not needed, doubles the surface
  during transition, and the shell-script-with-subcommands form is well-trodden
  (`git`, `cargo`, `kubectl`-style dispatch in shell). The Go binary would have
  to ship documentation for the same set of operations anyway.
- **`init --prod` writes the systemd unit and uses sudo internally.**
  Considered (and was the design in the first draft of this ADR). Rejected
  because mixing privilege levels inside a single command is a footgun: the
  rest of `init`'s output (`.env`, `rookery.toml`) should stay user-owned, and
  triggering `sudo` halfway through a script the operator just typed is the
  kind of surprise that breeds distrust. A separate `install` verb makes the
  privilege escalation an act the operator chooses, not a side effect of
  bootstrap.
- **Mode-detect `start`'s default from `/etc/systemd/system/rookery.service`
  existence.** Rejected: too magic. The systemd unit explicitly passes
  `--prod`, so the default only matters for operators typing `start` directly,
  which is rare and worth being explicit about.
- **Single `invite` verb instead of `invite create`.** Considered. Rejected
  because `invite list` and `invite revoke` are obviously-useful future verbs,
  and starting with the noun-verb form costs nothing while making the
  extension a non-breaking change.
- **`send-test` over `send-mail`.** The "test" suffix was honest about it
  being a dev-only tool, but `send-mail` is more general (smoke checks,
  manual reproduction, debugging a live instance's inbound path with a known
  message) and reads naturally even outside test contexts. We keep the
  `--encrypted` / `--fetch-key` flags so the dev-stack flow is unchanged.
- **A `Makefile` of `init`, `start`, `stop`, `invite create` targets.**
  Considered. Make's argument-handling is awkward (`make invite-create
  DAYS=7`) and Make is not a shell — multi-step recipes get clumsy fast. The
  dispatcher is a normal shell script with a `case "$1"`; this is a better
  fit.
- **Reverse the decision in ADR-0008 and add a web admin UI for invite
  creation.** Rejected for the same reasons ADR-0008 rejected it the first
  time. `rookery invite create` is one command; this is not the problem that
  needs a web UI.

## Migration notes

- `run.sh`, `scripts/new-invite.sh`, and `scripts/send-test-mail.sh` are
  deleted in the same change that adds `rookery`. There is no compatibility
  shim; operators following the README pick up `rookery` directly.
- The `./scripts:/scripts:ro` bind-mount on the postgres service in
  `compose.yaml` is removed.
- The systemd unit at `docs/ops/rookery.service` becomes a *template* used by
  `rookery init` to stage `./rookery.service` (with `User=` filled in from
  `--user` / `whoami`). Operators who installed the unit by hand under the
  old flow should `sudo rm /etc/systemd/system/rookery.service` and run
  `sudo rookery install` once to pick up the new path
  (`ExecStart=/opt/rookery/rookery start --prod`).
- `.gitignore` gains `rookery.service` (the staged unit, generated by
  `init`); `.env` and `Caddyfile` are already gitignored.
- PLAN.md §11.12 is rewritten to describe the subcommand surface above; this
  ADR is its long-form rationale.
- ADR-0005 and ADR-0008 gain a "superseded in part by ADR-0033" header note
  and their references to per-task scripts and to `docker compose exec` are
  updated to `rookery <subcommand>`. The shell-first principle in both ADRs is
  unchanged.
