# ADR-0015 — Authentication model: challenge/response login, cookie sessions, 2FA by design

**Status:** Accepted  
**Date:** 2026-05-16  
**Deciders:** project founders

---

## Context

rookery's key model (ADR-0010) stores the user's PGP private key only in a recovery
file the user holds. The server never holds the private key, a passphrase, or any
derivative of a passphrase. This has direct consequences for how authentication works:
there is no stored secret to verify against, only a stored public key.

---

## Decisions

### 1. Login: PGP challenge/response

Authentication is a two-round trip:

1. `GET /api/v1/auth/challenge?address=…` — server inserts a single-use nonce row
   (`auth_challenges` table) and returns `{ challenge_id, nonce }`. Challenges expire
   after 5 minutes and are single-use.
2. `POST /api/v1/auth/login` — client sends `{ address, challenge_id, signed_challenge }`
   where `signed_challenge` is an ASCII-armored detached PGP signature of the nonce,
   made with the user's private key. Server verifies the signature against the stored
   public key. On success, issues a session cookie.

The server holds no passphrase and no passphrase hash. The only authentication
credential on the server is the public key it already publishes via WKD.

### 2. Sessions: server-side cookie store

After a successful challenge/response, the server creates a session row in Postgres
and issues an HttpOnly, Secure, SameSite=Lax session cookie. Sessions use sliding
expiry: a session expires after `session_expiry_days` days of inactivity (default 7).
Logout deletes the session row server-side and clears the cookie.

The session token is never stored — only its SHA-256 hash. This means database read
access alone is insufficient to forge a session; an attacker also needs the raw
64-character hex token from the cookie.

Concurrent sessions are allowed. Sessions are listed in account settings by
`last_seen` timestamp only — no IP address or user agent is recorded by default
(pseudonymity property, §4 of PLAN.md). Operators who enable `log_connecting_ips`
in `rookery.toml` surface IP/UA on that same settings page so users can see exactly
what is stored.

### 3. 2FA by design, not bolt-on

The login flow is inherently two-factor: something you know (the passphrase) and
something you have (the recovery file). A stolen passphrase alone cannot authenticate
— signing the challenge requires the private key. A stolen file alone cannot
authenticate — the key stays locked without the passphrase.

This is a strictly stronger property than TOTP, which protects only the account
credential while doing nothing for message confidentiality. Here both factors gate
the private key itself. TOTP is therefore not implemented and not planned — it would
add complexity for no security gain under this model. WebAuthn / hardware-key unlock
is a Phase 7 candidate; it would replace the passphrase factor with a hardware-bound
one and is architecturally compatible with this ADR. SMS 2FA is ruled out (no phone
number, §5.4 of PLAN.md).

### 4. CSRF

The synchronizer token pattern. A per-session CSRF token is stored in the `sessions`
table and mirrored in a non-HttpOnly `rookery_csrf` cookie. Every state-changing
request must include it in the `X-CSRF-Token` header (API/fetch calls) or a `_csrf`
form field (plain HTML form POSTs). Compared with `subtle.ConstantTimeCompare`.

SameSite=Lax on the session cookie provides a first line of defence; the synchronizer
token is a second layer per §11.2 of PLAN.md.

### 5. Browser session key caching

After login the unlocked private key is wrapped with an AES-256-GCM key and stored
in `localStorage`, shared across all tabs and browser restarts. It is cleared on
logout. This caching is ergonomic, not cryptographic — both the wrapping key and the
wrapped blob live in `localStorage` together, so the protection against a same-origin
XSS is the same-origin policy and CSP, not the wrapping scheme. See ADR-0002 and
ADR-0016 for the mitigations (SRI, strict CSP).

---

## Alternatives considered

### Per-request PGP signing (stateless auth)

Rather than issuing a session cookie after login, every request could carry a
detached PGP signature of a per-request payload (e.g. `"METHOD /path timestamp"`),
eliminating the `sessions` table entirely. A breach would then leak zero
authentication material — the only server-side credential remains the public key.

**Why not now:** the server-rendered HTML architecture (ADR-0002) makes this
impractical for top-level browser navigations. The browser sends page requests
directly (link clicks, address bar, form submits that navigate) without JS
interception; there is no mechanism to attach a signature header to those requests
natively. Session cookies are the only standard mechanism for authenticating
browser-initiated navigations.

Per-request signing works cleanly for `fetch()`-based API calls from JS, since the
crypto module can sign those before sending. A hybrid model — session cookie for
page-load navigation, per-request PGP signature for API fetch calls — is coherent
and worth revisiting if the project ever adds a non-browser client path or moves
toward a heavier JS architecture. The `signChallenge` export in `web/crypto/index.js`
already provides the signing primitive needed for this.

**Replay mitigation if revisited:** the signed payload must include a monotonic
counter or a short-lived nonce to prevent replay. A server-side counter (one row
per user, not one row per session) would be far smaller than the current `sessions`
table while still preventing replay within the signing window.

### Passphrase-hash login (previous implementation)

The original implementation stored an Argon2id hash of the user's passphrase in
`users.login_hash` and verified it on every login. Removed in favour of
challenge/response because: (a) it contradicted the "server holds no authentication
material" security property; (b) a breach of `users.login_hash` gave an attacker
an offline cracking target; (c) it was redundant once challenge/response was
correctly implemented. The column was dropped in migration 0002.

### TOTP as a second factor

Considered and rejected. TOTP protects the account credential (preventing login
without the code) but does nothing to protect message confidentiality — an attacker
who steals the session cookie can still read all mail regardless of whether TOTP was
involved in login. The recovery-file model already provides a stronger second factor
that gates the private key itself, not just the login flow.

---

## Consequences

**Positive:**
- A full server breach leaks no material that would allow an attacker to authenticate
  as any user (no passphrase hashes, no private keys).
- Passphrase rotation is entirely client-side: re-export the recovery file with a
  new passphrase; no server state to update.
- The 2FA property is structural, not a UI feature that can be bypassed or disabled.

**Negative / trade-offs:**
- The `sessions` table exists and a breach exposes active session token hashes. An
  attacker who also intercepts a live session cookie could combine the two. Mitigated
  by storing only the SHA-256 hash (not the raw token) and by short sliding expiry.
- Stateless per-request signing — the stronger model — is deferred because the
  server-rendered architecture makes it impractical for page navigation today.
