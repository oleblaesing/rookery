# ADR-0007 — Custom domains are a v1 feature, with per-domain MTA-STS/TLS-RPT/ACME and a kill switch

**Status:** Accepted  
**Date:** 2026-05-15  
**Deciders:** project founders

---

## Context

The ability for a user to have `alice@alice.com` hosted on someone else's rookery
instance, while retaining full portability of that address, is the "killer feature"
that makes "deliberately replaceable" (ADR-0006) real at the user level. Without custom
domains, leaving an instance forces an address change. With them, it is a DNS update.

Custom domains are also the largest single workstream in the project (§8 Phase 3 of
PLAN.md). Getting the details right — per-domain DKIM, MTA-STS, WKD, ACME — is what
separates "Proton but you host it" from a genuine portable email identity story.

## Decision

**Custom domains are a v1 feature, shipped in Phase 3.** They are not optional and not
a Phase 7+ nice-to-have. The migration story (Phase 6) depends on them.

**Per-domain infrastructure:** every verified custom domain gets:
- Two DKIM keypairs (ed25519 + RSA-2048; see ADR-0024) generated server-side,
  encrypted at rest with the server master key (ADR-0023), published via DNS CNAMEs
  to per-instance selectors.
- WKD serving at `openpgpkey.<domain>` via CNAME to the instance's WKD endpoint,
  with per-domain TLS via ACME HTTP-01.
- MTA-STS policy at `mta-sts.<domain>` (CNAME + ACME HTTP-01). Mode `testing` for
  the first 48 hours after activation, then `enforce`. A per-domain kill switch
  (default off) can flip back to `testing` for incident response.
- TLS-RPT TXT record published; reports collected.
- DNS drift detection: the server continuously re-checks records and surfaces drift
  in the UI. A user whose registrar silently drops a CNAME finds out from rookery,
  not from a confused correspondent.

**Cert renewal failure is an unmissable alert.** MTA-STS cert expiry breaks inbound
mail for all users on a custom domain. The server raises an operator-visible alert
(log + Prometheus metric) at least 14 days before expiry. This is a hard requirement,
not a nice-to-have.

**ACME challenge type: HTTP-01.** DNS-01 is not used because it requires integrating
with registrar APIs, which would prevent the self-service setup flow from working
without the operator's involvement in every domain addition. HTTP-01 works because both
`openpgpkey.<domain>` and `mta-sts.<domain>` are reached via CNAME to our
infrastructure, so the HTTP-01 challenge is answered by our server.

**Custom domain addition is always user self-serve via the web UI.** It is a
load-bearing user-facing feature, not an operator shell operation. There is no
operator-only mode and no config toggle.

## Consequences

**Positive:**
- Users on custom domains have a portable email identity. Migration to a new instance
  is a DNS update, not an address change (Phase 6 makes this concrete).
- The "deliberately replaceable" promise (ADR-0006) holds at the user level.
- Per-domain MTA-STS means inbound mail to custom-domain users has transport security.

**Negative / trade-offs:**
- Phase 3 is the largest single workstream (≈6–8 weeks; §8 of PLAN.md). This is the
  cost of making portability real.
- MTA-STS cert expiry is a silent service failure if not caught. The 14-day alert
  requirement mitigates this but the operator must act on the alert.
- Per-domain DKIM key management (generation, storage, rotation) adds operational
  complexity. Rotation tooling (dual-selector transitions) is explicitly Phase 7.
- The CNAME-based WKD approach (CNAME → our infrastructure) means the user's domain
  cannot use the WKD direct method. This is acceptable because the advanced method is
  what we support (ADR-0024), and the CNAME keeps TLS simple.

## Alternatives considered

- **Custom domains as a Phase 7+ feature:** rejected. Without custom domains, migration
  (Phase 6) cannot be built as designed, and the "deliberately replaceable" claim is
  marketing rather than engineering.
- **DNS-01 ACME challenges:** rejected. Requires registrar API integration, which
  means either we integrate with dozens of registrars or the user must paste
  credentials into our UI. Neither is acceptable for a self-hostable product.
- **Operator-only domain addition, no self-serve flow:** rejected. Custom domains are
  a load-bearing user feature; requiring operator shell intervention for every domain
  addition contradicts the portability story and the "operator works from the shell,
  users work from the UI" principle.
- **Separate ACME/TLS infrastructure per domain (e.g. Caddy as a sidecar):** possible
  but adds a large dependency. certmagic (embedded in the Go server) handles this
  cleanly without a sidecar.
