# ADR-0001 — PGP-first mail server speaking standard SMTP; no new wire protocols

**Status:** Accepted  
**Date:** 2026-05-15  
**Deciders:** project founders

---

## Context

rookery is a self-hostable mail server aimed at making PGP end-to-end encryption the
default for users who consent to it. The project needs to decide how rookery instances
communicate with the rest of the email world and with each other.

The temptation in any "secure messaging" project is to invent a new protocol: a custom
federation layer, a DHT for key discovery, a libp2p gossip network, or a blockchain
anchored key transparency log. Each of these would make rookery instances a closed
network rather than mail servers.

## Decision

rookery is a mail server. It speaks standard SMTP (RFC 5321) to the rest of the world
and to other rookery instances. There are no novel wire protocols, no inter-instance
coordination layer, no special protocol between rookery servers, no DHT, no libp2p,
no blockchain.

PGP end-to-end encryption uses OpenPGP (RFC 9580) formatted as PGP/MIME (RFC 3156) —
the same format `gpg --encrypt | mutt` produces. Key discovery uses WKD (draft-koch-
openpgp-webkey-service) and Autocrypt (later). DKIM, SPF, DMARC, MTA-STS, and TLS-RPT
are used exactly as the standards specify.

The cryptographic libraries are `ProtonMail/go-crypto` (server-side) and `OpenPGP.js`
(browser-side). We are consumers of these libraries; we do not implement cryptographic
primitives ourselves.

## Consequences

**Positive:**
- Any RFC-conforming mail client (Thunderbird, Apple Mail, Gmail, Proton) can exchange
  mail with a rookery instance.
- A PGP key generated by rookery is a normal OpenPGP key; the user can import it into
  `gpg` or Thunderbird at any time.
- Two rookery instances communicate via exactly the same mechanism that Postfix and
  Exchange use: MX lookup + SMTP. Nothing special is needed.
- The threat model is known and documented, not invented.

**Negative / trade-offs:**
- SMTP metadata (From, To, Subject, timestamps, Received chain) is visible to
  intermediaries. This is a property of email, not a rookery failure. We say so plainly
  in onboarding and in the threat model (§4 of PLAN.md). Metadata-resistance is
  permanently out of scope.
- Subject lines are not encrypted. The PGP ecosystem (RFC 3156, Memory Hole, RFC 9152)
  does not have interoperable encrypted-Subject support. Users who need subject privacy
  use a generic placeholder. We document this and move on.
- IP addresses of correspondents are visible in the `Received:` chain. This is the
  SMTP norm; it is not a property we can change without leaving SMTP.

## Alternatives considered

- **Custom federation protocol between rookery instances.** Rejected. Two rookery
  instances are just two mail servers; SMTP is the already-deployed federation protocol
  for email. Adding a custom layer would make rookery a closed network and undermine
  the "deliberately replaceable" principle (ADR-0006).
- **Matrix, Activity-Pub, or similar.** Rejected. Not email. The project's premise is
  that email is worth fixing, not replacing.
- **Implementing OpenPGP from scratch.** Rejected. `ProtonMail/go-crypto` and
  `OpenPGP.js` are actively maintained, production-tested, and interop-tested.
  Cryptographic implementations must be audited at scale; ours would not be.
