# ADR-0008 — One in-app role (user); no admin in the app; invite-only registration; first user via new-invite.sh

**Status:** Accepted  
**Date:** 2026-05-15  
**Deciders:** project founders

---

## Context

Every multi-user web application must answer: who can do what, and how is that access
controlled? The common answer is a role system (admin vs user, or fine-grained RBAC).
rookery's answer is different, and the difference is deliberate.

## Decision

**There is one in-application role: user.** There is no `is_admin` column, no admin
role flag, no admin session, no admin-only web routes. Every authenticated user of the
web application is equal in terms of application-level permissions.

**The operator is defined by having shell access** to the VPS running the instance.
The operator's tools are:
- The shell and `psql` for reading and writing the database directly.
- The scripts in `/opt/rookery/scripts/` for common operations.
- The config file (`rookery.toml`) for instance-wide policy.

**Registration is invite-only, always.** Invite tokens are rows in the `invites`
table. The operator creates them via `./scripts/new-invite.sh` (which does a `psql
INSERT`). There is no open-registration mode; the abuse risk on a pseudonymous,
IP-log-free instance is not a trade-off worth offering as a config knob.

**The first user is created exactly like every other user.** The operator runs
`./scripts/new-invite.sh`, visits the resulting URL in a browser, picks a local-part,
generates their key, downloads the recovery file, and lands in an empty inbox. There is
no "bootstrap admin" dance, no separate setup flow, no special first-run login.

**User-issued invites** are a Phase 7+ option. In v1, only the operator creates invites.

## Consequences

**Positive:**
- The application has no privileged role to exploit. A compromised user session can
  harm only that user's mailbox; it cannot administer the instance.
- The access control model is small enough to fully audit: every endpoint is either
  unauthenticated, authenticated-user-required, or internal-port-only. There are no
  admin-only routes to misplace.
- The data model is free of role columns that create complexity (e.g., "what happens
  when the admin user deletes themselves?").
- The operator model is honest: the person with `root` on the box can do anything to
  the database; adding an in-app admin role would be security theatre layered on top of
  that reality.

**Negative / trade-offs:**
- Operators who want a co-admin with shell-free access to instance management cannot do
  so in v1. This is an explicit constraint (P10 of PLAN.md); the target operator is a
  single competent Linux user.
- Invite management is a shell operation, not a web UI action. For small instances (the
  primary use case) this is fine. If a public instance grows large enough to need a web
  UI for invite management, that is a signal the instance should split or a v2
  conversation.
- There is no in-app moderation queue; abuse signals surface via Prometheus metrics.

## Alternatives considered

- **Admin role (is_admin boolean or role enum):** rejected. Creates an in-app
  privileged surface that is both less capable than shell access and an additional
  attack surface. The operator already has root; reinventing access controls in the
  browser is busywork (P10 of PLAN.md).
- **Admin role restricted to account-management operations only:** the appeal is
  delegating invite creation to a trusted non-operator. Deferred to Phase 7+ (user-
  issued invites would cover most of this without a role system). Not a v1 need.
- **Open registration (opt-in or default):** rejected. The abuse risk on a
  pseudonymous, IP-log-free instance is not a trade-off worth offering at all.
  Invite-only is the only mode.
- **Magic-link bootstrap (no scripts):** considered. A one-time-use bootstrap URL
  printed to the server log on first run would reduce friction. Rejected because it adds
  a special first-run code path; `./scripts/new-invite.sh` is already 30 lines and
  works the same way every time, including the first.
