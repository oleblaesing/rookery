# ADR-0005 — Operator UX: shell-first, no admin web UI, 30-minute time-to-instance NFR

**Status:** Accepted  
**Date:** 2026-05-15  
**Deciders:** project founders

---

## Context

rookery aims to be self-hostable by "one competent person" without becoming a full-time
email sysadmin (§1 of PLAN.md). This person — the operator — already has root on the
VPS and is comfortable with a shell, a text editor, and `docker compose`. The question
is what interface the operator uses to administer the instance.

## Decision

**The operator's interface is the shell.** There is no admin web UI, no in-app admin
role, no setup wizard. Every operator action is either:
- Editing the config file (`rookery.toml`) and restarting the container.
- Running a documented shell script from `/opt/rookery/scripts/` via `docker compose exec`.
- Running a documented `psql` query.

**The 30-minute active-configuration-time NFR** is: a competent Linux user with a
domain and a properly-provisioned VPS (PTR set, port 25 not blocked by provider) goes
from "fresh box" to "first invite URL generated" in under 30 minutes of active work.
DNS propagation and PTR turnaround are explicitly excluded from the budget because they
are outside our control. The README measures this honestly.

**First-run bootstrapping is automatic.** On first run, the server generates the
database schema, primary-domain DKIM keypairs, and the ACME account, and logs the DNS
records the operator needs to publish. No interactive setup flow; no web wizard.

**Operator scripts** are thin shell wrappers around `psql` (see §11.12 of PLAN.md):
`new-invite.sh`, `suspend-user.sh`, `unsuspend-user.sh`, `delete-user.sh`,
`print-dns.sh`, `print-deliverability-stats.sh`, `rotate-master-key.sh`.
Custom domain management is handled by users via the web UI, not by operator scripts.

**There is no `--admin` flag, no `is_admin` column in the database, and no operator
concept inside the application.** The operator is defined by having `root` on the box.

## Consequences

**Positive:**
- The operator model is simple enough to fully understand. No separate access control
  layer to secure or audit inside the application.
- Shell scripts are inspectable before running (`cat /opt/rookery/scripts/new-invite.sh`
  prints 30 lines of obvious `psql`). Operators are never asked to trust a black box.
- The data model constraint — every operator-meaningful state change must be a small,
  invariant-preserving row update — is a beneficial design pressure.
- No admin web UI means no admin web UI to secure, maintain, or exploit.

**Negative / trade-offs:**
- Non-shell-comfortable operators cannot use rookery. This is an explicit constraint
  (P10, §2 of PLAN.md); the target operator is a "competent Linux user."
- There is no green-tick DNS propagation checker built into the UI. The server logs
  DNS status; the operator reads logs. For an audience that reads logs, this is fine.
- Instances cannot be managed remotely over the web by a non-technical co-owner of the
  instance. Phase 7+ could add a read-only metrics API but there will never be an
  in-app admin role.

## Alternatives considered

- **Admin web UI with a separate admin role:** rejected. Adds a privileged access
  surface that must be secured, maintained, and audited. For a small instance, the
  operator already has root; reinventing access controls in a browser is busywork (P10).
- **Separate admin CLI binary:** rejected. The operator scripts are simpler and more
  transparent as shell scripts than as a compiled binary with its own argument parser
  and error handling.
- **Ansible / Terraform integration:** not our job. We ship a working container; the
  operator integrates it into their toolchain as they see fit.
- **SaaS-style hosted control panel:** explicitly rejected (P10); no phone-home, no
  telemetry, no hosted anything.
