# ADR-0036 — Per-domain DKIM: CNAME-based key sharing for custom domains

**Status:** Accepted
**Date:** 2026-05-23
**Deciders:** project founders

---

## Context

DKIM-signed outbound mail requires the sending domain's DKIM public key to be published
in DNS so receiving MTAs can verify signatures. For the primary domain, the server
generates a keypair at startup and the operator publishes TXT records directly. For
custom domains (Phase 4), we need an approach that:

- Does not require the instance operator to manage per-user DNS records.
- Does not require the user to copy long base64 TXT values (RSA-2048 public keys are
  ~392 bytes in base64 — bulky, error-prone to transcribe).
- Works correctly with the signing infrastructure that already uses `dkim_keys` per domain.

### Two approaches

**A. Per-domain TXT records:** generate unique keypairs per custom domain (already
supported by the `dkim_keys` schema), and ask the user to publish two TXT records
(`ed25519-1._domainkey.<custom>` and `rsa-1._domainkey.<custom>`) with the base64 public
key values.

*Pros:* cryptographic isolation — a key compromise on one domain does not affect others.
*Cons:* users must transcribe two long TXT values; RSA-2048 public key in base64 is ~392
chars, pushing the limit of some DNS providers' TXT record input fields (255-byte per-
string limit requires string splitting, which many web UIs handle opaquely). More
operational complexity per domain.

**B. CNAME to primary (shared keys):** the instance's primary DKIM keys are the only keys
that need to exist. Custom domains publish a CNAME record pointing their DKIM selector at
the primary domain's TXT record:

```
rookery._domainkey.alice.com  CNAME  rookery._domainkey.<primary>
```

Receiving MTAs look up `rookery._domainkey.alice.com`, follow the CNAME, and retrieve the
TXT value at `rookery._domainkey.<primary>`. Verification succeeds because the signature
was produced with the corresponding private key.

*Pros:* the user publishes a CNAME, not a long TXT value; consistent with the CNAME
approach used for `openpgpkey.*` and `mta-sts.*` (ADR-0034 / ADR-0024). Key rotation
affects all custom domains simultaneously (one operation updates the primary TXT). No
per-domain keypairs to generate or store.
*Cons:* shared key — if the instance's primary DKIM private key is compromised, all
custom domains on the instance are affected until the key is rotated.

---

## Decision

**Use CNAME-based key sharing (Approach B).** This matches §5.2b of PLAN.md and the
CNAME-first philosophy of the rest of Phase 4's DNS record set.

### Selector names

The primary domain's DKIM selectors are `rookery-ed25519` and `rookery-rsa`. Both
existing and custom domain outbound mail uses these selectors. The `d=` field in the
DKIM-Signature header is set to the envelope From domain (e.g. `alice.com`); the `s=`
field is `rookery-ed25519` or `rookery-rsa`.

The CNAME the user publishes:

```
rookery._domainkey.alice.com  CNAME  rookery._domainkey.<primary>
```

(Note: only one CNAME is needed because both selectors are served from the same TXT
record at the primary domain. Receiving MTAs follow the CNAME for each selector lookup
independently. Both `rookery-ed25519._domainkey.alice.com` and
`rookery-rsa._domainkey.alice.com` CNAME to the respective primary TXT records.)

Correction: two CNAMEs are needed, one per selector:

```
rookery-ed25519._domainkey.alice.com  CNAME  rookery-ed25519._domainkey.<primary>
rookery-rsa._domainkey.alice.com      CNAME  rookery-rsa._domainkey.<primary>
```

The DNS wizard in the UI shows both records.

### Signing implementation

`dkim.Sign()` currently queries `dkim_keys` by domain name. For custom domains, no
`dkim_keys` row exists (they share the primary's key). The implementation change:

When `Sign(ctx, domainName, r)` finds zero active keys for `domainName`, it falls back to
loading keys for the **primary domain** (the domain where `domains.is_primary = TRUE`).
The `SignOptions.Domain` is still set to the custom domain (`alice.com`) — this is
correct: the `d=` tag in the DKIM-Signature must match the From header's domain.

No new `dkim_keys` rows are written for custom domains. The primary domain's existing
rows are reused at signing time.

### Key rotation impact

When the primary DKIM keys are rotated (Phase 8 tooling), the operator publishes the new
TXT records at the primary domain's selectors. Because custom domains point to the primary
via CNAME, they automatically pick up the new key after DNS propagation. The operator does
not need to touch each custom domain's DNS during rotation — this is a significant
operational advantage over per-domain TXT records.

### Isolation trade-off

The shared key means a primary DKIM private key compromise (from server breach) affects
all custom domains simultaneously. This is already the case for every other instance-level
secret (session key, master key). The server master key (ADR-0023) is the first line of
defence; the DKIM private key is encrypted at rest with it. The compromise risk is the
same as compromising any instance-level credential, and the mitigation (backup the master
key, rotate on suspicion) is the same.

Per-domain key isolation (Approach A) can be adopted in a future phase if demand arises —
the `dkim_keys` schema already supports it. Migration path: generate per-domain keypairs,
publish TXT records, update `Sign()` to prefer per-domain keys over the primary fallback.
This is a backwards-compatible schema change.

---

## Consequences

**Positive:**
- User publishes a short CNAME, not a 392-byte base64 TXT value.
- Consistent with the CNAME philosophy of the Phase 4 DNS record set.
- Key rotation (Phase 8) is a single operation that covers all custom domains.
- No new keypairs to generate, store, or manage per custom domain.

**Negative / trade-offs:**
- Shared key — primary key compromise affects all custom domains. Accepted for Phase 4.
- DKIM signature verification shows `d=alice.com` but the key is at
  `rookery-ed25519._domainkey.<primary>` via CNAME. Some strict DMARC evaluators or
  forensics tools may note the CNAME indirection. This is not a compliance issue — CNAME
  resolution is required by DKIM verifiers per RFC 6376 §3.6.2.

## Alternatives considered

- **Per-domain TXT records (Approach A):** rejected for Phase 4 on UX grounds (long TXT
  values) and operational simplicity. Preserved as a Phase 8+ upgrade path.
- **Per-domain sub-labels at the primary domain (e.g. `alice-com-ed25519._domainkey.<primary>`):**
  rejected. Requires the instance operator to maintain per-user TXT records in their own
  DNS zone, which breaks the user-self-service model and couples the instance operator's
  DNS management to every new custom domain addition.
- **Single shared selector with all custom domains pointing to one TXT:** this is
  effectively what the CNAME approach does — each custom domain selector CNAMEs to the
  corresponding primary selector. The final TXT lookup is the same for all domains.
