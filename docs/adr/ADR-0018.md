# ADR-0018 — Reserved local-parts: auto-creation on domain verification

**Status:** Accepted
**Date:** 2026-05-23
**Deciders:** project founders

---

## Context

Several email local-parts are conventionally or RFC-mandatorily reserved on every domain:
`postmaster`, `abuse`, `hostmaster`, and `webmaster`. Operators of the primary domain
and users who verify custom domains must have these addresses functional, both to satisfy
RFC compliance and because DMARC aggregate reports often deliver to `dmarc@` or
`abuse@` addresses configured by the user.

The existing schema already has a `reserved_local_parts` table seeded with these four
values. This ADR records the auto-creation behaviour that links them to actual addresses
at domain verification time.

---

## Decision

### Which local-parts are reserved

`postmaster`, `abuse`, `hostmaster`, and `webmaster` — matching RFC 2142 and the DMARC
aggregate report convention. No others. The list is fixed in code and in the
`reserved_local_parts` table; it is not operator-configurable.

### When reserved addresses are created

On every domain that becomes managed by the instance (primary domain at startup; custom
domains at successful verification), the application inserts alias rows in the `addresses`
table for each of the four reserved local-parts. The aliases point to the domain's owner:

- **Primary domain:** owner is the first user created on the instance. If no user exists
  yet (first-boot race), reserved addresses are created for the primary domain the first
  time `EnsureKeys` runs after the first user registers.
- **Custom domain:** owner is `domains.owner_user_id` at verification time.

Insertion is idempotent (`INSERT ... ON CONFLICT DO NOTHING`). Re-running domain
verification or restarting the server does not create duplicates.

### Address row format

Each reserved address is inserted as:

```
local_part:      "postmaster" (etc.)
address:         "postmaster@<domain>"
user_id:         <owner user ID>
domain_id:       <domain ID>
is_alias:        TRUE
alias_target_id: <owner's primary_address_id>
```

The alias points to the owner's `primary_address_id` at insertion time. If the owner
later changes their primary address, the alias target is **not** automatically updated —
it continues to deliver to the address that was primary at domain-verification time.
Operators who want to reroute reserved addresses can update the `alias_target_id` column
via `rookery psql`.

### User cannot delete reserved addresses

The application rejects DELETE requests for addresses whose `local_part` is in the
`reserved_local_parts` table. The delete handler checks this before executing. The UI
omits the delete button for reserved addresses.

### User cannot register a reserved local-part as a regular address

The registration and address-creation handlers check `reserved_local_parts` before
inserting. Attempting to register `postmaster@alice.com` (for example) returns a `409`
with a clear error message ("this local-part is reserved and cannot be used as a regular
address").

---

## Consequences

**Positive:**
- RFC 2142 compliance is automatic — operators do not have to manually create these
  addresses.
- DMARC aggregate reports addressed to `abuse@<domain>` or `postmaster@<domain>` reach
  the domain owner without extra configuration.
- Consistent behaviour across primary and custom domains: every domain managed by this
  instance has functioning required addresses.

**Negative / trade-offs:**
- The owner's primary address receives all postmaster/abuse/etc. mail without
  filtering. On a custom domain with a catch-all (ADR-0017), this is acceptable — the
  owner expects all such mail. On a high-traffic domain this could be noisy; Phase 8 can
  add optional forwarding targets.
- If the domain owner changes their primary address after domain verification, the
  reserved alias targets do not update. Documented as a known limitation; the operator
  can fix it via `rookery psql`.

## Alternatives considered

- **Configurable reserved addresses:** rejected. RFC 2142 is a compliance requirement,
  not a preference. The operator might accidentally remove `postmaster@` and break their
  MTA-STS compliance. Hardcoding the four addresses is the right default.
- **Route reserved addresses to a configurable fallback email (e.g. instance contact
  email):** rejected. The address must exist as a local mailbox for correct SMTP handling
  (bounces, auto-replies, DMARC reports). An external forward would require outbound SMTP
  at alias-resolution time, which is unnecessary complexity.
- **Create reserved addresses only on the primary domain, not custom domains:** rejected.
  An operator or user who brings `alice.com` to the instance expects `postmaster@alice.com`
  to work. External senders rely on these addresses existing on every domain they interact
  with.
