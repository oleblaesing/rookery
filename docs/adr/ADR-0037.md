# ADR-0037 — MTA-STS lifecycle for custom domains

**Status:** Accepted
**Date:** 2026-05-23
**Deciders:** project founders

---

## Context

MTA-STS (RFC 8461) lets a domain publish a policy over HTTPS that tells sending MTAs to
require TLS when delivering mail. ADR-0024 committed the high-level decisions:
`testing` mode for 48 hours after activation, then `enforce`; per-domain kill switch via
`mta_sts_mode` in the database; cert expiry alert ≥14 days before expiry.

This ADR records the implementation details that ADR-0024 left open: the `id=` field
semantics, the policy endpoint serving mechanism, and the cert-expiry alerting approach
given that Caddy (not the Go binary) manages certificates.

---

## Decision

### Policy endpoint location and serving

MTA-STS requires the policy to be served at:

```
https://mta-sts.<domain>/.well-known/mta-sts.txt
```

The `mta-sts.<domain>` hostname is pointed at the rookery instance via CNAME (ADR-0034).
Caddy provisions the cert (ADR-0035). The Go HTTP server receives the request and
dispatches based on the `Host` header:

The handler for `/.well-known/mta-sts.txt` inspects the request `Host`. If the host is
`mta-sts.<something>`, it extracts `<something>`, looks up the domain in the database,
and renders the policy. If the domain is not found or not verified, it returns 404.

No new routing infrastructure is needed: the existing `Host`-inspection pattern already
used for WKD (`handleWKDKey` checks the domain in the URL path) extends naturally here.
The primary domain's MTA-STS endpoint also uses this handler (it is not special-cased).

### Policy format

```text
version: STSv1
mode: <testing|enforce|none>
mx: mail.<primary-domain>
max_age: <seconds>
```

- `mode: testing` during the first 48 hours and when the kill switch is active.
- `mode: enforce` after the testing window expires and the kill switch is not set.
- `mode: none` is not served (if the domain has `mta_sts_mode = 'disabled'`, the endpoint
  returns 404 so no policy is published).
- `max_age`:
  - `testing`: 86400 (24 hours) — short TTL so senders do not cache a broken config.
  - `enforce`: 604800 (7 days) — reasonable TTL, low enough to recover from a key outage
    within a week if the kill switch is activated.
- `mx: mail.<primary>` — the instance's primary MX hostname. Custom domains all route
  inbound via the primary instance's MX record, so this value is always the same.

### The `id=` field

The `_mta-sts.<domain>` TXT record contains `v=STSv1; id=<id>`. The `id` signals to
caching MTAs that the policy has changed and must be re-fetched.

- The `id` is a 16-character hex string generated at domain verification time and stored
  in `domains.mta_sts_id`.
- The `id` is **regenerated** when:
  1. The mode transitions from `testing` to `enforce` (the background worker updates the
     column when it flips the mode).
  2. The operator sets `mta_sts_mode = 'disabled'` and back to `null` (kill-switch cycle).
- The `id` is **not** regenerated on server restart or other events (unnecessary churn).
- **The user must re-publish the `_mta-sts.<domain>` TXT record** when the `id` changes.
  The domain detail page shows the current required `id` value. After the
  `testing → enforce` transition, the UI shows a banner: "Your MTA-STS policy is now in
  enforce mode. Update the `_mta-sts.<domain>` TXT record with the new ID: `<new-id>`."
  The drift checker (ADR-0038) will catch the mismatch if the user does not update the
  record, and the status panel will show a drift warning.

### Mode transition: testing → enforce

A background goroutine runs at startup and ticks every 15 minutes. For each domain where:

```sql
mta_sts_mode IS NULL         -- NULL means "follow the automatic schedule"
AND mta_sts_mode_changed_at < now() - interval '48 hours'
AND verified_at IS NOT NULL
```

it updates:

```sql
UPDATE domains
SET mta_sts_mode = 'enforce',
    mta_sts_mode_changed_at = now(),
    mta_sts_id = <new-hex-id>
WHERE id = <domain-id>
```

The transition is idempotent (once set to 'enforce', the condition no longer matches).

**Mode state in the database:**
- `NULL` — automatic (testing for 48h, then enforce). This is the post-verification
  default. The Go code interprets NULL as "compute mode from age."
- `'testing'` — forced testing mode (kill switch activated by operator via psql).
- `'enforce'` — enforce mode (set by the background worker after 48h, or manually).
- `'disabled'` — kill switch: policy endpoint returns 404; no MTA-STS policy published.

The policy serving code maps these to the rendered `mode:` line:
- `NULL` and age < 48h → `mode: testing`
- `NULL` and age ≥ 48h → `mode: enforce`  
- `'testing'` → `mode: testing`
- `'enforce'` → `mode: enforce`
- `'disabled'` → 404

### Cert expiry alert

Caddy manages certificates; the Go binary has no direct access to cert expiry dates.
The alerting approach:

A background goroutine runs once per day. For each verified domain, it makes a TLS dial
to `mta-sts.<domain>:443` with a 10-second timeout and inspects the leaf certificate
from `tls.ConnectionState().PeerCertificates`. If the leaf cert expires within 14 days,
the goroutine emits:

```
slog.Warn("mta-sts cert expiry approaching",
    "event_key", "mta_sts_cert_expiry_warning",
    "domain", domain,
    "expires_at", cert.NotAfter,
    "days_remaining", daysLeft,
)
```

This is a structured WARN log at a stable `event_key` so operators can alert on it with
log-scraping tools (grep, Loki/Grafana, etc.). Phase 5 will promote this to a Prometheus
metric (consistent with ADR-0024's "becomes a Prometheus metric in Phase 5" note).

TLS dial errors (refused, no cert, timeout) are logged at DEBUG level with
`event_key = "mta_sts_cert_check_failed"` — they typically indicate the domain has not
yet received its first on-demand cert from Caddy and are not actionable until cert
issuance has succeeded.

## Consequences

**Positive:**
- Host-header dispatch is consistent with WKD serving — one pattern for all virtual-host
  HTTP serving in rookery.
- The `testing → enforce` background worker is a simple background goroutine, no external
  scheduler needed.
- The `id=` semantics are minimal: only regenerated when the policy changes, not on every
  server restart. Unnecessary TXT-record churn for users is avoided.
- The cert expiry check via TLS dial works regardless of how Caddy stores cert metadata.

**Negative / trade-offs:**
- When the `testing → enforce` transition fires, the user must update the `_mta-sts`
  TXT record with the new `id`. If they do not, the drift checker (ADR-0038) will detect
  the mismatch and show a warning, but inbound mail is not dropped (senders cache the
  old policy for its TTL duration).
- The TLS dial for cert expiry checking adds a daily network connection per custom domain.
  For a public instance with hundreds of custom domains, this is a brief burst of outbound
  TLS connections once per day — acceptable, not a concern in v1.
- The 15-minute tick on the `testing → enforce` worker means the actual transition could
  lag by up to 15 minutes past the 48-hour mark. This is not a correctness issue.

## Alternatives considered

- **Trigger testing→enforce immediately at T+48h via a one-shot timer per domain:**
  considered. Rejected for simplicity — a polling worker avoids timer bookkeeping and
  handles server restarts correctly. The 15-minute lag is immaterial.
- **`id=` as a Unix timestamp (changes on every policy change):** considered. Rejected
  because timestamps change every time the server restarts or any non-policy field is
  updated. A stable hex ID that changes only on policy changes is cleaner and less likely
  to trigger unnecessary re-fetches by caching MTAs.
- **Caddy Admin API for cert expiry data:** considered. Caddy's `/pki/ca/...` endpoint
  does not directly expose cert expiry for individual ACME-provisioned hostnames. Parsing
  the Caddy cert storage directory is fragile. TLS dial is more robust.
- **Serve `mta-sts.txt` from Caddy config directly (static file):** rejected. MTA-STS
  policy content is dynamic (mode changes, `id` changes); it must be generated by the Go
  application on demand.
