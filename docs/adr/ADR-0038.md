# ADR-0038 — DNS preflight and drift detection

**Status:** Accepted
**Date:** 2026-05-23
**Deciders:** project founders

---

## Context

DNS records can drift after a domain is verified: a registrar silently removes a record,
a user accidentally edits a CNAME, a TTL-cached wrong value takes over. On a mail server,
drifted DNS can mean: inbound mail stops arriving (MX removed), DKIM verification starts
failing (CNAME removed), MTA-STS enforcement causes senders to drop mail (MTA-STS cert
cert revoked but mode still enforce). The user may not notice for days.

Phase 4 builds the initial domain verification check (ADR-0034). This ADR covers the
continuous background re-checking that runs after a domain is verified.

---

## Decision

### What is checked

For each verified domain, the checker validates:

| Record | Type | Expected |
|---|---|---|
| `<domain>` | MX | `mail.<primary>` with any preference value |
| `<domain>` | TXT (SPF) | value starts with `v=spf1 include:_spf.<primary>` |
| `rookery-ed25519._domainkey.<domain>` | CNAME | points to `rookery-ed25519._domainkey.<primary>` |
| `rookery-rsa._domainkey.<domain>` | CNAME | points to `rookery-rsa._domainkey.<primary>` |
| `openpgpkey.<domain>` | CNAME | points to `openpgpkey.<primary>` |
| `mta-sts.<domain>` | CNAME | points to `mta-sts.<primary>` |
| `_mta-sts.<domain>` | TXT | value is `v=STSv1; id=<domains.mta_sts_id>` |

The `_rookery-challenge` TXT record is **not** checked after verification — it is
intentionally ephemeral and users are told they can remove it.

The DMARC and TLS-RPT TXT records are **not** validated by the drift checker. Their exact
values are under user control (different `rua` addresses are valid); checking them would
produce false drift alerts. They are shown in the domain setup UI as advisory records.

### Check cadence

A single background goroutine processes all verified domains. The default check interval
is **3600 seconds (1 hour)**, configurable via `dns_check_interval_seconds` in
`rookery.toml`. The goroutine iterates domains in creation-order; if there are many
domains, they are processed in batches to spread DNS queries over the interval.

The goroutine uses `context.WithTimeout(ctx, 10s)` per individual DNS lookup. Transient
DNS failures (SERVFAIL, timeout) update `dns_status` to `unknown` rather than `drifted`
— a single failure is not drift, it is noise. Three consecutive `unknown` results for the
same record are logged as a WARNING with `event_key = "dns_lookup_persistent_failure"`.

### Storage

A single `dns_status` JSON column on the `domains` table stores the latest check result
as a map of record → status:

```json
{
  "MX": "ok",
  "SPF": "ok",
  "DKIM_ED25519_CNAME": "drifted",
  "DKIM_RSA_CNAME": "ok",
  "WKD_CNAME": "ok",
  "MTA_STS_CNAME": "ok",
  "MTA_STS_TXT": "unknown"
}
```

Individual record values (expected vs actual) are not stored persistently — they are
fetched fresh by the domain detail page when the user views it. The `dns_status` JSON is
used for the status chip (all-ok / partially-drifted / drifted) visible in the domain list.

`domains.dns_last_checked_at` is updated on every check run (even if no change). This
lets the UI show "Last checked: 47 minutes ago."

### UI surfacing

**Domain list (settings page):** each domain row shows a small status chip:
- ✓ green — all records OK
- ⚠ yellow — one or more `unknown` (transient failure)
- ✗ red — one or more `drifted`

**Domain detail page:** the per-record status panel (same panel used during initial
verification) shows the latest check result with record name, type, expected value, actual
value (from a fresh DNS lookup triggered on page load), and a ✓ / ✗ / ? indicator. A
"Re-check now" button triggers an immediate DNS lookup for all records of this domain and
refreshes the fragment.

The per-record fresh lookup on page load is triggered by a `partials.js` `swap()` call to
`GET /api/v1/domains/{id}/dns-status` (HTML fragment). This is a **one-shot** fetch on
page open — not a polling loop. Continuous auto-polling is not done post-verification
(unlike the initial verification flow) because drift is a slow, infrequent event;
polling every 5 seconds after verification would waste DNS queries and server resources.

### Logging

When the background check finds a drifted record:

```
slog.Warn("DNS drift detected",
    "event_key", "dns_drift_detected",
    "domain", domain,
    "record", "DKIM_ED25519_CNAME",
    "expected", "rookery-ed25519._domainkey.primary.example",
    "actual", "",  // empty = NXDOMAIN
)
```

One log line per drifted record per check run. If drift persists across multiple check
runs, the same line is emitted each run (no suppression). Operators using log aggregation
can deduplicate on `(domain, record)` if needed.

### Implementation

Go stdlib: `net.LookupMX`, `net.LookupTXT`, `net.LookupCNAME`. No external DNS library.
Lookups use the system resolver (which on Docker uses the container's `/etc/resolv.conf`,
pointing to Docker's internal DNS). For production correctness, the check should use a
public resolver to see what the rest of the internet sees; `rookery.toml` can optionally
configure a `dns_check_resolver` address (default: `"9.9.9.9:53"`) which is passed to a
`net.Resolver` with a custom `Dial` function. Quad9 is the default because it is
privacy-respecting (no query logging by default) and consistent with rookery's
pseudonymity stance.

---

## Consequences

**Positive:**
- Operators (and users) learn about DNS drift from the dashboard, not from a confused
  correspondent reporting delivery failures.
- The JSON status column is compact and query-friendly (no separate per-record table,
  no complex schema).
- One-shot page-load fetch (not continuous polling) keeps the UI load light.
- Configurable resolver lets operators point at an authoritative or public resolver to
  get an "outside view" of their DNS, not Docker's internal view.

**Negative / trade-offs:**
- With a 1-hour check cadence, drift is detected at most 1 hour after it occurs. For MTA-STS
  in enforce mode, a drifted CNAME can mean inbound mail drops for up to 1 hour before the
  operator sees the alert. The 14-day cert expiry warning (ADR-0037) gives a much longer
  window for the worst failure mode.
- Storing the status as a JSON blob rather than normalized rows means querying for "how
  many domains have a drifted DKIM CNAME?" requires a JSON expression, not a simple
  `WHERE`. For Phase 4's small scale this is fine; Phase 8+ can normalize if needed.
- The drift checker does not auto-remediate (it cannot — DNS is not under the server's
  control). It only surfaces the problem.

## Alternatives considered

- **Separate `domain_dns_checks` table with one row per record per check run:** rejected.
  Unnecessary normalization complexity for a status value that only needs the latest state.
  Historical check data is in the logs.
- **Continuous per-domain polling (same as verification flow):** rejected. Drift is slow;
  1-hour background checks are sufficient and much cheaper than per-user polling loops.
- **Alerting via email to the domain owner:** considered. Deferred — the structured WARN
  log is the v1 mechanism. Phase 8 could add an in-app notification or email alert if
  operators request it.
- **Using the system resolver only (no configurable external resolver):** rejected. Docker's
  internal DNS returns internal addresses for some names; a mail server needs to verify what
  the public internet sees. Defaulting to `9.9.9.9:53` (Quad9) and making it configurable
  is the right posture.
