# ADR-0035 — Per-domain ACME via Caddy: on-demand TLS for custom domains

**Status:** Accepted
**Date:** 2026-05-23
**Deciders:** project founders

---

## Context

Phase 3 established Caddy as the TLS sidecar (ADR-0024). The primary domain and
`openpgpkey.<primary>` are provisioned via a static `Caddyfile` written by `rookery init`.
That works for one domain; Phase 4 introduces an unbounded number of custom domains, each
requiring TLS certificates for:

- `mta-sts.<custom-domain>` (serves the MTA-STS policy file over HTTPS, required by RFC 8461)
- `openpgpkey.<custom-domain>` (serves WKD keys over HTTPS, required by the WKD spec)

We need a mechanism to provision and auto-renew these certs without requiring the operator
to re-run `rookery init` or manually edit the Caddyfile every time a user adds a domain.

### Options evaluated

**A. Regenerate Caddyfile + reload Caddy** — When a domain is verified, rookery rewrites
the Caddyfile and sends `caddy reload` (via a signal or the Caddy API's reload endpoint).
Pros: simple, matches the "static config" model. Cons: requires the Go server process to
write to a mounted filesystem path and signal a sibling container. File-write + signal
composition is brittle under concurrent domain additions; a partial write during reload is
a reliability risk.

**B. Caddy Admin API (dynamic config)** — rookery calls Caddy's JSON config API
(`POST /config/apps/http/servers/…`) to add each new domain. Pros: fully dynamic.
Cons: in-memory Caddy config diverges from the Caddyfile on disk; a restart resets
to the static file and drops all dynamically-added domains. Requires either a "persist
on restart" reconciliation routine in rookery or a Caddy config-store backend (Postgres),
neither of which is worth the complexity for v1.

**C. Caddy on-demand TLS with an `ask` authorization endpoint** — Caddy provisions a
cert the first time it receives a TLS handshake for an unknown hostname, but only if a
pre-configured `ask` URL returns HTTP 200 for that hostname. rookery exposes a small
internal endpoint that returns 200 for `mta-sts.<verified-domain>` and
`openpgpkey.<verified-domain>`, 403 otherwise. Certs are cached in the `caddy-data`
volume and auto-renewed. The Caddyfile is written once by `rookery init` and never
modified at runtime.
Pros: fully automatic after initial setup, no runtime file writes, restart-safe (certs
are in the volume, Caddy re-authorizes and auto-renews on startup). Cons: on-demand TLS
has slightly different cert-issuance timing (cert is provisioned on first real request,
not at domain-activation time); rate-limit abuse risk is mitigated by the `ask` gate.

---

## Decision

**Use Caddy on-demand TLS (Option C).** The `ask` authorization endpoint is the correct
tool for this use case — it exists precisely to authorize cert issuance for hostnames not
known at Caddy startup.

### `rookery` internal TLS-ask endpoint

```
GET /internal/tls-ask?domain=<hostname>
```

- **No authentication.** This endpoint is on the Docker-internal network only (not
  reachable from outside the container network). Caddy calls it; external callers cannot.
- **Response 200:** the hostname is `mta-sts.<verified-domain>` or
  `openpgpkey.<verified-domain>` where `<verified-domain>` is a domain row with
  `verified_at IS NOT NULL` in the database.
- **Response 403:** anything else, including the primary domain's hostnames (those use
  static Caddyfile blocks, not on-demand TLS) and unverified or unknown domains.
- **Response 503:** database unavailable; Caddy will retry.

The endpoint does a single `SELECT id FROM domains WHERE domain = $1 AND verified_at IS NOT NULL`
(O(1) indexed lookup). No caching needed at the application layer; Caddy rate-limits
`ask` calls internally.

### Caddyfile changes

`rookery init` generates a Caddyfile with the following addition alongside the existing
primary-domain blocks:

```caddyfile
{
    on_demand_tls {
        ask http://rookery:8080/internal/tls-ask
        interval 1m
        burst 5
    }
}

# Catch-all for custom-domain hostnames (mta-sts.*, openpgpkey.*)
:443 {
    tls {
        on_demand
    }
    reverse_proxy rookery:8080
}
```

The primary domain blocks remain static (no `on_demand` flag) and take priority over the
catch-all `:443` block in Caddy's matching order.

### Cert issuance timing

A cert for `mta-sts.alice.com` is provisioned when the first HTTPS request to that
hostname arrives. For MTA-STS, this is typically the first time a sending MTA checks the
policy (which happens during SMTP session setup to `alice.com`). There is a window of a
few minutes between domain verification and the first cert issuance during which `mta-sts`
returns a TLS error. This is acceptable: MTA-STS is already in `testing` mode for 48
hours post-verification, so no inbound mail is dropped during this window.

If an operator wants to pre-warm certs immediately after domain verification, they can
`curl -s https://mta-sts.<domain>/.well-known/mta-sts.txt` — Caddy will provision the
cert synchronously on that first request.

### Let's Encrypt rate limits

Let's Encrypt allows 50 certs per registered domain per week and 5 failed validations
per hour per hostname. On-demand TLS is rate-limited by Caddy's `interval`/`burst`
config (1 new cert attempt per minute, burst of 5). For a small instance with a few
custom domains, this is well within limits. For a public instance with many users adding
domains rapidly, operators should monitor Let's Encrypt rate limit headers; this is
documented in the operator runbook.

### Restart behaviour

After a Caddy restart, existing certs are already in the `caddy-data` volume. Caddy
auto-renews them without calling the `ask` endpoint again (renewal uses the cached ACME
account and cert data). The `ask` endpoint is only called for initial issuance.

---

## Consequences

**Positive:**
- Zero runtime Caddyfile modifications. The operator generates it once with `rookery init`;
  all subsequent custom domains work automatically.
- Restart-safe: cert data persists in the Docker volume, not in application memory.
- The `ask` endpoint is a simple DB query — easy to reason about and test.
- Consistent with ADR-0024's principle of "no certmagic in Go, no new Go dependency."

**Negative / trade-offs:**
- Cert issuance is demand-triggered (on first TLS connection), not proactive. There is a
  window of seconds to minutes between domain activation and first cert issuance. The 48h
  MTA-STS testing window makes this safe for MTA-STS; WKD similarly has no hard-fail mode.
- On-demand TLS requires port 80 to be open for HTTP-01 challenge responses — same
  requirement as the primary domain (already in the operator runbook).
- The internal `/internal/tls-ask` endpoint must not be accessible from outside the Docker
  network. The compose file binds rookery only to the internal network; this is already
  the production topology for other Caddy→rookery proxying.

## Alternatives considered

(See Options A and B in the context above.)

- **`certmagic` back in the Go binary:** rejected. ADR-0024 removed it; re-adding it
  would be a regression. `certmagic` requires Postgres-backed cert storage, an ACME
  goroutine, and port 80 management inside the binary — all of which Caddy already handles.
- **Wildcard cert for `*.{primary}`:** rejected. Let's Encrypt wildcard certs require
  DNS-01, which requires registrar API access (explicitly ruled out in ADR-0024). Also,
  `mta-sts.alice.com` is not a subdomain of the primary instance domain, so a wildcard on
  the primary domain would not help.
- **Self-signed certs for `mta-sts.*` and `openpgpkey.*`:** rejected. Both WKD and MTA-STS
  require valid (CA-signed) TLS. Self-signed certs would cause spec-non-compliance.
