# ADR-0032 — Spam filtering: rspamd + Redis, ClamAV opt-in

**Status:** Accepted
**Date:** 2026-05-25
**Deciders:** project founders
**Supersedes:** nothing (new decision)
**Required before:** Phase 5 spam-filter code lands (per §13 of PLAN.md)

---

## Context

Inbound mail needs basic spam filtering. The requirements from PLAN.md §11.13 and
§10 are:

- "Reasonable out-of-the-box behaviour for a small instance" — not Gmail-quality,
  just good defaults.
- Must fit the 2 GB RAM budget for a single-user instance on a $5 VPS.
- ClamAV is opt-in; enabling it adds ~1 GB RAM, which not every operator has.
- No per-user Bayesian training UI in v1 — that's Phase 8+ work.
- Spam goes to Trash, not a separate Spam folder (keeps the mailbox model flat,
  per §11.5).
- Operator tunes rspamd via mounted volume if they want to; we don't expose
  rspamd-tuning subcommands in the dispatcher.

The operator target is a technically competent individual running an instance
for themselves, family, or a small group — not a mail service provider needing
enterprise-grade anti-spam.

## Decision

### Spam filter: rspamd

**rspamd** is the spam filter for rookery.

Alternatives evaluated:

| Alternative | Rejected because |
|---|---|
| SpamAssassin | Perl runtime, slow, heavy; memory cost on a 2 GB VPS is prohibitive alongside the rest of the stack. |
| milter-based (e.g. OpenDKIM + custom) | Not a spam filter; just policy enforcement. Does not classify. |
| Hand-rolled heuristics in Go | Not worth the maintenance burden. rspamd's rule set is large, actively maintained, and free. |
| No spam filtering | Invite-only registration and per-user rate limits limit blast radius, but inbound spam still needs filtering. An instance without any spam filtering cannot realistically be used for correspondence. |

rspamd ships stock defaults that provide:
- Header and content heuristics
- RBL (real-time blackhole list) queries
- SPF / DKIM / DMARC alignment checks
- URL reputation
- Neural network scoring module (on by default; pre-trained weights ship with rspamd)
- Fuzzy hash matching against known spam corpora

These work out of the box with no tuning. The thresholds rspamd ships with
(`reject` ≥ 15, `add header` ≥ 6) are well-calibrated and we do not override them.

### Backing store: Redis (bundled sidecar)

rspamd requires Redis for: bayes data, fuzzy hashes, rate limiting, and several
other modules. Redis is added as a sidecar container in `compose.yaml` alongside
rookery and postgres. Operators do not interact with Redis directly; they do not
see it as a separate operational concern.

Redis was already implied by rspamd's requirements. The only design question was
whether to use an external Redis or bundle one. We bundle it: the 30-minute
active-setup-time NFR (§10) makes "provision a separate Redis instance" a
non-starter, and the RAM footprint of Redis at this scale is negligible (~30 MB
idle).

### ClamAV: opt-in behind a compose profile

ClamAV adds ~1 GB RAM when its signature database is loaded. That pushes a
2 GB VPS instance to the edge; the §10 RAM budget calls ClamAV out explicitly
as opt-in.

Implementation: ClamAV runs as a compose service behind `--profile clamav`.
rspamd is always configured to try clamd at `clamav:3310`; when ClamAV is not
running, rspamd's antivirus module logs a warning and continues without virus
scanning. This avoids requiring two rspamd configurations.

Operators who want virus scanning:
1. Start the stack with `docker compose --profile clamav up` (or add `clamav`
   to their active profiles in the systemd unit's `ExecStart` flags).
2. ClamAV starts and rspamd discovers it automatically on the next message.

No `rookery.toml` flag is needed for ClamAV — the compose profile IS the toggle.

### Integration: rspamd HTTP check API in the inbound SMTP path

rookery's Go inbound SMTP handler calls rspamd's HTTP check API
(`POST rspamd:11333/checkv2`) after reading the raw message but before storing
it. The call is **soft-fail**: if rspamd is unreachable or times out, the
message is delivered anyway with a warning log. This prevents rspamd startup
lag from blocking inbound mail.

rspamd action → SMTP response mapping:

| rspamd action | rookery behaviour |
|---|---|
| `no action` | Deliver normally |
| `add header` (medium confidence) | Deliver; rspamd adds `X-Spam-*` headers to the message |
| `rewrite subject` | Deliver (rspamd rewrites the subject header) |
| `greylist` | SMTP 4xx (soft reject); sender should retry |
| `soft reject` | SMTP 4xx (soft reject); sender should retry |
| `reject` (high confidence) | SMTP 5xx; message never enters storage |

The rspamd URL is configured in `rookery.toml` as `spam.rspamd_url`, defaulting
to `http://rspamd:11333` (the bundled container). Setting it to `""` disables
spam checking entirely (for testing or custom setups).

### What this does NOT include in v1

- **No Bayesian training.** No "mark as spam" button, no `learn_spam` / `learn_ham`
  feedback loop. rspamd's stock neural network and heuristics do plenty without
  per-instance bayes training. Per-user bayes training is Phase 8+.
- **No separate Spam folder.** Spam goes to Trash (§11.5). `X-Spam-*` headers are
  preserved and visible in the message detail view.
- **No rspamd admin UI.** Operators who want to tune rspamd access it via
  `rookery exec rspamd rspamc ...` or mount a volume over `/etc/rspamd/local.d/`.
  The dispatcher does not add rspamd-specific subcommands.
- **No whitelist/blacklist management UI.** Operators manage these via rspamd's
  own mechanisms (local.d overrides, maps).

## Consequences

**Positive:**

- Reasonable out-of-the-box spam filtering with zero operator tuning required.
  Stock rspamd defaults are actively maintained by the rspamd project; we inherit
  improvements on each image update.
- Memory budget met: rspamd + Redis add roughly 200–400 MB idle, leaving 1.4–1.8 GB
  for rookery, postgres, and headroom. ClamAV's 1 GB is opt-in.
- Soft-fail integration means rspamd startup or temporary unavailability does not
  block inbound mail.
- No new Go dependencies. The rspamd HTTP check uses `net/http` (stdlib).
- ClamAV as a compose profile is truly opt-in: operators who don't need it pay
  zero cost.

**Negative / trade-offs:**

- rspamd is a non-trivial additional container. Operators who inspect the compose
  file see three services they did not have before (redis, rspamd, and optionally
  clamav). We mitigate this by making Redis and rspamd invisible in normal
  operation — they are not listed in `rookery help` and require no operator action.
- IP reputation is a separate concern (§9.1, §10). rspamd's RBL queries help
  with blocking known-bad senders; they do nothing for the *outbound* IP reputation
  problem that new instances face with Gmail. That problem is documented in
  `docs/ops/spam-runbook.md` rather than solved here.
- Stock rspamd defaults will misclassify edge cases — false positives (legitimate
  mail in Trash) and false negatives (spam in inbox). This is accepted and
  documented. Operators who need tighter control mount a volume over
  `/etc/rspamd/local.d/` and tune rspamd directly.
- "No Bayesian training in v1" is a real limitation. A fresh rspamd instance
  has no instance-specific spam corpus. The pre-trained neural network and global
  RBL checks compensate; per-user bayes is Phase 8+.

## Alternatives considered

**Mail-in-a-box / mailinabox-style bundled stack.** Rejected: brings in Postfix,
Dovecot, Roundcube, and a full Ubuntu stack — the antithesis of rookery's
single-binary design. We use rspamd as a sidecar, not as the center of the stack.

**Milter interface instead of HTTP.** rspamd supports both. HTTP check API is
simpler to integrate from Go (standard net/http, no milter protocol library needed)
and easier to reason about (request/response, not a long-lived connection).
P12 (aggressive dependency minimalism) tilts toward the simpler interface.

**Only use rspamd's local socket.** Docker compose services communicate over the
internal network by service name; HTTP over TCP is the natural interface and
avoids bind-mount complexity for a Unix socket.

**redis:alpine vs valkey vs KeyDB.** Redis 7 Alpine is the most boring, most
widely deployed choice and matches what rspamd's documentation targets. We stay
on Redis 7; if Redis's licensing trajectory becomes a problem (the SSPL change
was in Redis 7.4), operators can swap the image name in compose.yaml to Valkey
without any configuration change.
