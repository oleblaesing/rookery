# ADR-0003 — No persistent device state, no account recovery

**Status:** Accepted  
**Date:** 2026-05-15  
**Deciders:** project founders

---

## Context

rookery's key model (ADR-0010) stores the user's PGP private key only in a recovery
file the user exports at signup. The server never holds the private key, and
nothing is persisted in the browser between sessions. This has direct consequences
for the "new device," "second login on the same device," and "lost passphrase"
scenarios that every key management system must address.

## Decision

**Every login on every device requires the recovery file.**

The recovery file is the second factor. After the server validates the passphrase and
issues a session cookie, the login page shows a recovery-file import form — there is no
bypass. The user selects their `.asc` file; the key is unlocked in memory with the
passphrase, verified against the account fingerprint, and cached in `localStorage` for
the session. Nothing is written to IndexedDB or any other persistent browser store.
`localStorage` is cleared on logout; the key does not survive explicit logout, but
it does persist across page navigations and browser restarts within the login session.

This means there is no distinction between "a device I have used before" and "a new
device." Every login on any device is the same: passphrase plus recovery file. The
account is not bound to any device; the device has no memory of past logins. Using
the account from a second laptop, a borrowed machine, or a new browser is identical
to using it from the first.

**There is no account recovery.** If the user loses their recovery file or their
passphrase, their encrypted mail is unrecoverable — by them, by the server operator,
by anyone. The server has nothing to surrender. We say this plainly and repeatedly in
onboarding (§5.2a step 7 of PLAN.md). There is no "forgot my passphrase" link,
anywhere, ever.

**Multi-device access is inherently supported.** Any device can use the account given
the passphrase and recovery file. There is no sync step because there is nothing
device-specific to synchronise: the session key is cleared on logout and the
recovery file is device-independent.

## Consequences

**Positive:**
- The server cannot be compelled to surrender private keys it does not possess.
- Offline brute-force attacks on private-key material are impossible against the server
  (there is nothing to exfiltrate).
- The threat model is simple: the user owns their key, the user is responsible for
  their recovery file.
- No "encrypted blob on server" attack surface (see ADR-0010 for that analysis).

**Negative / trade-offs:**
- Lost recovery file + lost passphrase = permanent mail loss. This is the correct
  behaviour given the system's design, but it imposes real UX burden on users.
  Mitigation: the onboarding flow makes the loss scenario as clear as possible
  (§5.2a step 7, §5.4 of PLAN.md). The target audience (techies who understand
  keypairs, §2 P8) is more likely to appreciate the trade-off than to be surprised
  by it.
- Every login requires the recovery file. For the target audience (people who understand
  keypairs and keep their recovery file accessible, §2 P8) this is acceptable and
  expected. For non-technical users it is a barrier — but those users are not the v1
  target (§5.4 of PLAN.md).
- Key rotation is harder when the user holds the only copy of the old key. The rotation
  attestation protocol (ADR-0028) handles the "notify correspondents" side; the "new
  key generation" side happens in the browser exactly as it did at signup.

## Alternatives considered

- **IndexedDB caching of the encrypted key across logouts:** considered and rejected.
  Storing the encrypted private key in IndexedDB would mean subsequent logins only
  require the passphrase — the recovery file would not be needed on a device that has
  logged in before. This quietly weakens the 2FA property: anyone who knows the
  passphrase and has access to the browser can log in without the recovery file. The
  recovery file is the second factor; making it optional after first use undermines the
  point. The current `localStorage` scheme caches the unlocked key within the login
  session (persisting across page navigations and browser restarts) but clears it on
  logout, so every new login still requires both passphrase and recovery file.
  Rejected: every login must require both.
- **Server-side encrypted key blob (e.g. Proton's model):** considered seriously. The
  advantages: login is passphrase-only (server provides the blob). The disadvantages:
  (a) the server becomes a centralised brute-force target for offline key cracking;
  (b) the "server hands client a substituted blob on login" MITM is now a real attack
  vector; (c) the server holds key-shaped material it can be legally compelled to
  produce. Rejected in favour of client-only custody. Documented in ADR-0010.
- **Shamir Secret Sharing / social recovery:** interesting, but adds significant
  protocol complexity. Belongs in a future ADR if the community wants it. Not v1.
- **Hardware key (YubiKey) for primary storage:** appealing. Deferred to Phase 7
  (§8 of PLAN.md) because it requires WebAuthn / CTAP2 browser support to be used as
  a storage medium (not just a second factor). Conceptually compatible with this ADR.
