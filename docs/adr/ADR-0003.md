# ADR-0003 — Single-device, no account recovery

**Status:** Accepted  
**Date:** 2026-05-15  
**Deciders:** project founders

---

## Context

rookery's key model (ADR-0010) stores the user's PGP private key only in the user's
browser (IndexedDB) and in a recovery file the user downloads at signup. The server
never holds the private key. This has direct consequences for the "new device" and
"lost passphrase" scenarios that every key management system must address.

## Decision

**Single-device is the default; additional devices require a manual import step.**

When a user logs in on a new browser or device, the server returns a session cookie
but cannot provide the private key — it does not have it. The browser checks IndexedDB,
finds nothing, and the UI prompts the user to import their recovery `.asc` file. After
the user drags in the recovery file and enters their PGP passphrase, the key is
decrypted in memory. If the user ticks "remember on this device," the encrypted blob is
written to IndexedDB for that browser origin.

**There is no account recovery.** If the user loses both their recovery file and their
PGP passphrase, their encrypted mail is unrecoverable — by them, by the server
operator, by anyone. The server has nothing to surrender. We say this plainly and
repeatedly in onboarding (§5.2a step 7 of PLAN.md). There is no "forgot my passphrase"
link, anywhere, ever.

**There is no multi-device key sync.** The same key can be loaded on multiple devices
by importing the recovery file once on each device, but there is no automatic
synchronisation mechanism. This is deliberate: sync requires putting the key somewhere
both devices can reach, which either re-introduces server-side key storage or requires
a novel sync protocol. Both options are rejected.

## Consequences

**Positive:**
- The server cannot be compelled to surrender private keys it does not possess.
- Offline brute-force attacks on private-key material are impossible against the server
  (there is nothing to exfiltrate).
- The threat model is simple: the user owns their key, the user is responsible for
  their recovery file.
- No "encrypted blob on server" attack surface (see ADR-0010 for that analysis).

**Negative / trade-offs:**
- Lost recovery file + lost passphrase = permanent mail loss. This is the correct
  behaviour given the system's design, but it imposes real UX burden on users.
  Mitigation: the onboarding flow makes the loss scenario as clear as possible
  (§5.2a step 7, §5.4 of PLAN.md). The target audience (techies who understand
  keypairs, §2 P8) is more likely to appreciate the trade-off than to be surprised
  by it.
- Setting up a second device is a manual step (import recovery file). For the target
  audience this is acceptable. For non-technical users it is a barrier — but those
  users are not the v1 target (§2 P8, §5.4 of PLAN.md).
- Key rotation is harder when the user holds the only copy of the old key. The rotation
  attestation protocol (ADR-0028) handles the "notify correspondents" side; the "new
  key generation" side happens in the browser exactly as it did at signup.

## Alternatives considered

- **Server-side encrypted key blob (e.g. Proton's model):** considered seriously. The
  advantages: second-device setup is simpler (server has the blob); passphrase changes
  only need to re-encrypt one blob. The disadvantages: (a) the server becomes a
  centralised brute-force target for offline key cracking; (b) the "server hands client
  a substituted blob on login" MITM is now a real attack vector (a malicious server can
  replace the blob with one it knows the key for); (c) the server holds key-shaped
  material that it can be legally compelled to produce, even if it can't crack it today.
  Rejected in favour of client-only custody. Documented in ADR-0010.
- **Shamir Secret Sharing / social recovery:** interesting, but adds significant
  protocol complexity. Belongs in a future ADR if the community wants it. Not v1.
- **Hardware key (YubiKey) for primary storage:** appealing. Deferred to Phase 7
  (§8 of PLAN.md) because it requires WebAuthn / CTAP2 browser support to be used as
  a storage medium (not just a second factor). Conceptually compatible with this ADR.
