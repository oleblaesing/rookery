# ADR-0039 — Per-user data export and import (instance portability)

**Status:** Accepted
**Date:** 2026-05-28
**Deciders:** project founders

---

## Context

A self-hostable mail server is only truly self-hostable if users can leave it. Phase 6/7
of the plan included per-user export and instance migration in the roadmap. This ADR
describes the design chosen for standalone delivery ahead of those phases.

The core security invariant — **the server never holds a user's PGP private key** — shapes
the entire design. Any approach that requires the server to decrypt the archive, or to hold
both the archive and the key at the same time, violates this invariant and is rejected.

The operator-side `rookery backup`/`restore` (shell-script level, backs up the whole
instance including the Postgres dump) has a different audience (operators) and threat model
(disaster recovery, not user portability). This ADR addresses only the user-facing path.

---

## Decision

### Four key decisions

**1. Archive encryption: PGP-encrypted to the user's own public key.**

The server assembles the archive in a background goroutine and encrypts it to the user's
active public key using `ProtonMail/go-crypto`. The archive is binary OpenPGP format (not
ASCII-armored), usable as:

```
gpg -d rookery-archive-*.tar.gpg | tar x
```

Only the holder of the corresponding private key can decrypt it. This means:

- An instance-B server cannot decrypt an archive from instance A. (It doesn't need to —
  the browser does it.)
- The archive can be stored anywhere (cloud storage, a private URL) without exposing the
  user's mail to third parties.
- The encrypted archive can be published with a time-limited bearer token because it is
  opaque to anyone without the private key.

**2. Export is async; download URL delivered to inbox.**

Large mailboxes could take seconds to minutes to export. A synchronous HTTP response would
time out. The export is triggered with `POST /api/v1/users/me/export`, which starts a
background goroutine and returns `{"status":"pending"}` immediately. The goroutine writes
the archive to disk, then sends a system inbox message containing the download and migration
links. The browser polls `GET /api/v1/users/me/export/status` to show progress.

**3. Archive download URL is a 24-hour bearer token, publicly accessible, CORS-enabled.**

`GET /export/<token>` requires no session cookie. Security comes from the 24h
expiry, the 256-bit entropy of the token, and the PGP encryption (even if the URL leaks,
the downloader cannot read the contents without the private key). After the first successful
download the job is marked `downloaded`; the file is still served until expiry to allow
retry. After expiry the file is deleted by the hourly cleanup worker.

The endpoint adds `Access-Control-Allow-Origin: *` so instance-B browsers can issue
cross-origin fetch requests to it.

**4. Import happens on a logged-out migration page that creates the account; instance B
proxies the ciphertext.**

Import is **not** an action taken by an already-logged-in user. The original design put the
import form in the logged-in settings page and required the importing user's active-key
fingerprint to equal the archive manifest's fingerprint. That breaks the actual migration
case: on a new instance the user has no account, or a freshly-generated key that does not
match — so the fingerprint check always failed. (See the "Logged-out migration page"
section below for the superseding design.)

Import flow on instance B (the destination), all on the unauthenticated `/migrate` page:
1. User provides the recovery file + passphrase for their OLD account, the archive URL, an
   invite token for instance B, and a desired username.
2. The browser unlocks the private key, then instance B's server fetches the encrypted
   bytes from instance A and streams them back (`GET /api/v1/import/fetch?url=…`,
   unauthenticated because no account exists yet).
3. The browser decrypts using OpenPGP.js. Successful decryption proves the recovery key is
   the key the archive was encrypted to — and therefore matches the manifest fingerprint.
4. The browser registers a new account with the archive's own public key (derived from the
   private key): `POST /api/v1/users/register` (invite-gated; sets session + CSRF cookies).
5. The browser POSTs the plaintext tar to `POST /api/v1/users/me/import`. The fingerprint
   ownership check now passes by construction, since the account was created with the
   archive's key.
6. The browser caches the unlocked key and navigates to the inbox (auto-login).

Why B proxies (not the browser fetching A directly): same-origin policy would prevent the
browser from directly fetching from instance A. Having B proxy keeps the CORS story simple
(all browser requests are same-origin to B).

The proxy endpoint validates the target URL against SSRF attacks: HTTPS only, hostname must
not resolve to loopback or RFC-1918 private addresses (checked at DNS resolution time and
again at TCP dial time to resist DNS rebinding).

Power users with very large archives can use the CLI escape hatch:
```
gpg -d rookery-archive-*.tar.gpg | \
  curl -X POST --data-binary @- \
       -H "Content-Type: application/octet-stream" \
       -H "X-CSRF-Token: <token>" \
       https://instance-b/api/v1/users/me/import
```

### Archive format

```
manifest.json
public_key.asc
known_keys.json
drafts.json
messages.json
message_attachments.json
blobs/<sha256>.eml
blobs/<sha256>.eml
...
```

**manifest.json** fields: `schema_version`, `exported_at`, `source_instance`,
`key_fingerprint`, `display_name`, `message_count`, `blob_count`, `known_key_count`,
`draft_count`, `addresses`, `custom_domains`.

`schema_version` guards against future incompatibilities; import rejects unknown versions.

**Exported:** `user_keys` (active public key only), `messages` (all rows for the user,
all folders), raw `.eml` blobs (content-addressed, deduped by sha256), `message_attachments`
metadata, `known_keys`, `drafts`, addresses and custom domains (manifest metadata only,
not recreated on import).

**Never exported:** `sessions`, `invites`, `dkim_keys`, `login_hash` (dropped in migration
0002 anyway), `totp_secret`, `used_bytes`, `quota_bytes`, any other user's data.

### Security properties

- **Ownership check on import:** `manifest.key_fingerprint` must match the importing user's
  active key fingerprint. A user cannot import someone else's archive.
- **Path traversal guard:** all tar entry names are validated; entries with `..` or absolute
  paths are rejected.
- **Blob integrity:** each blob's SHA-256 is recomputed from the actual bytes and verified
  against the tar entry name. Tampered blobs are rejected.
- **Schema version guard:** archives with an unknown `schema_version` are rejected.
- **Entry count limit:** import enforces a 1,000,000 entry maximum.
- **SSRF on proxy endpoint:** URL must be HTTPS; target IP checked at DNS resolution time
  and at dial time.

### Database table

```sql
CREATE TABLE export_jobs (
    id          UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID        NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    token_hash  TEXT        NOT NULL UNIQUE,
    status      TEXT        NOT NULL DEFAULT 'pending'
                CHECK (status IN ('pending', 'ready', 'downloaded', 'expired', 'failed')),
    file_path   TEXT,
    error_msg   TEXT,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    expires_at  TIMESTAMPTZ NOT NULL DEFAULT now() + INTERVAL '24 hours',
    ready_at    TIMESTAMPTZ
);
```

`token_hash` stores `SHA-256(raw_token)` so the raw token only ever exists in the user's
inbox message. A full database dump does not allow downloading archives.

### New server package: `internal/archive/`

`ExportUser(ctx, db, st, userID, domain, w)` — streams the encrypted tar to w.
`ImportUser(ctx, db, st, userID, r)` — reads plaintext tar from r and ingests.

Uses only stdlib (`archive/tar`, `encoding/json`, `crypto/sha256`) and the already-present
`ProtonMail/go-crypto` dependency. No new Go dependencies.

### Logged-out migration page

Import lives on the destination instance's unauthenticated `/migrate` page
(`handleMigratePage`, `migrate.js`). Because the account is created *from the archive's own
key* (step 4 above), the manifest-fingerprint ownership check in `ImportUser` passes by
construction — there is no longer any need to register separately with a matching key
first. Account creation stays **invite-gated**: the operator's control over who joins the
instance is preserved.

Ordering is fetch → decrypt → register → import, so a bad URL, passphrase, or archive fails
*before* the invite is consumed or an account is created. If register succeeds but the
import POST fails, the decrypted tar is kept in browser memory and the page offers a "retry
import" action (the user is logged in by then; there is no settings-based import to fall
back on).

The archive-fetch proxy `GET /api/v1/import/fetch` is unauthenticated for this flow (the
account does not exist when it runs). It retains all SSRF guards. It is functionally an
open proxy to arbitrary *public* HTTPS hosts — already true for any logged-in user under
the old authenticated route; a request rate limit is a possible follow-up.

### Migration deep-link

The inbox notification contains two links:

1. Direct download: `https://instance-a.example/export/<token>`
2. Migration deep-link: `https://instance-a.example/migrate?archive=<url-encoded-download-url>`

The user replaces `instance-a.example` in the deep-link with their new instance's address
and opens it (no login required). The `/migrate` page pre-fills the archive URL from
`?archive=` (and the invite token from `?invite=` when present); the user supplies their
recovery file, passphrase, invite token, and chosen username.

### Addresses and custom domains

The manifest records addresses and custom domains for reference only. Import does **not**
recreate them — the user re-establishes them via the normal domain/address flows on instance
B. Message headers reference original addresses verbatim; this is correct (the address
existed at send time on instance A).

### Relation to operator-side backup/restore

`rookery backup`/`restore` is operator-facing, backs up the whole instance at the filesystem
level, and requires the operator's passphrase. The per-user export/import described here is
user-facing, scoped to a single mailbox, and uses PGP encryption to the user's own key.
They are complementary: an operator can restore an entire instance from `rookery backup`;
a user can move their personal mailbox with the export/import flow.

## Consequences

**Positive:**
- Users can leave at any time with no operator involvement.
- Archives are encrypted to the user's own key — standard backup hygiene applies.
- The `gpg` interop recipe means the format is not opaque to technical users.
- No new Go dependencies.
- Async export + inbox delivery avoids HTTP timeouts on large mailboxes.

**Negative / trade-offs:**
- Import decrypts in the browser — the full plaintext tar is momentarily in browser memory.
  For a 5 GiB mailbox this would be a problem; the 25 MiB per-message limit means in
  practice archives are bounded by the number of messages × metadata + total attachment size.
  The current implementation buffers the decrypted bytes before POSTing (streaming request
  bodies are not universally supported; noted as a future optimisation).
- Drafts are not deduplicated on re-import (no natural key). A second import will insert
  duplicate draft rows. Documented; acceptable for the expected use case (one migration).
- The proxy endpoint adds a network hop (B → A) that the browser could otherwise make
  directly (with a CORS preflight). The tradeoff is simpler CORS handling and no changes
  to instance A's CORS policy.

## Alternatives considered

- **Server decrypts the archive for import:** rejected. Violates the core invariant.
- **Browser fetches from A directly:** rejected. Requires CORS on instance A's auth-protected
  API; complicates the instance-A setup and creates upgrade ordering requirements.
- **Synchronous export:** rejected. HTTP timeouts on large mailboxes; poor UX.
- **Export to S3/object storage:** out of scope for v1; the `data/exports/` directory on the
  instance is the simplest store that satisfies the 24-hour retention requirement.
- **ZIP instead of tar:** rejected. `tar` has better streaming support and is the standard
  for Unix data archives. `gpg | tar x` is a familiar idiom.
