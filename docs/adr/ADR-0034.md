# ADR-0034 — Custom-domain verification challenge protocol

**Status:** Accepted
**Date:** 2026-05-23
**Deciders:** project founders

---

## Context

Before the server activates a custom domain (generates DKIM keys, enables WKD, starts
serving MTA-STS), it must verify that the user controls the domain. Without verification,
a malicious user could register `gmail.com` and receive mail intended for Gmail users.

The verification mechanism must be:
- Self-service (no operator involvement per §5.2b / ADR-0008).
- DNS-based (no HTTP redirect, no email confirmation — both have awkward bootstrapping
  issues when the domain is not yet pointed at this instance).
- Usable without registrar API access (consistent with ADR-0024's HTTP-01 / no-DNS-01 stance).
- Partial-progress friendly (user can publish records incrementally and come back later).

---

## Decision

### Verification token

A cryptographically random 32-byte value encoded as URL-safe base64 (no padding). The
token is generated by `crypto/rand` and stored in `domains.verification_token` when the
user submits the "add domain" form. An expiry is stored in `domains.verification_expires_at`
(7 days from generation).

On expiry, the token is regenerated automatically the next time the user opens the domain
detail page (idempotent: new token, new 7-day window, same domain row). The user re-publishes
the new value.

### DNS record for verification

The user publishes:

```
TXT _rookery-challenge.<domain>  "<token>"
```

This record is checked first during verification polling. It is the gating record: until
the challenge TXT is present, no other records are activated.

The `_rookery-challenge` prefix is instance-controlled and collision-free with standard
DNS record names (no RFC uses this prefix).

### Full DNS record set shown in the UI

The domain setup wizard shows all records the user must publish before the domain is fully
functional. Records are shown together so the user can publish them in one DNS-provider
session rather than returning repeatedly:

| Record | Type | Value |
|---|---|---|
| `alice.com` | MX (10) | `mail.<primary>` |
| `alice.com` | TXT | `v=spf1 include:_spf.<primary> ~all` |
| `rookery._domainkey.alice.com` | CNAME | `rookery._domainkey.<primary>` |
| `_dmarc.alice.com` | TXT | `v=DMARC1; p=quarantine; rua=mailto:dmarc@alice.com` |
| `openpgpkey.alice.com` | CNAME | `openpgpkey.<primary>` |
| `mta-sts.alice.com` | CNAME | `mta-sts.<primary>` |
| `_mta-sts.alice.com` | TXT | `v=STSv1; id=<mta_sts_id>` |
| `_smtp._tls.alice.com` | TXT | `v=TLSRPTv1; rua=mailto:tls-reports@alice.com` |
| `_rookery-challenge.alice.com` | TXT | `<token>` |

The `rua` addresses in DMARC and TLS-RPT are pre-filled with the user's registered address
on the domain; the user can change them before publishing.

### Verification state machine

```
pending → checking → verified
                  ↘ partial   (challenge present, not all records yet)
                  ↘ failed    (challenge absent after expiry)
```

**Checking** is triggered by the user clicking "Check now" or by the background polling
endpoint. The server does a DNS lookup sequence:

1. Resolve `_rookery-challenge.<domain>` TXT → must match `domains.verification_token`.
2. Resolve `alice.com` MX → must point to `mail.<primary>`.

If both pass, the domain moves to `verified`. Records 3–9 (DKIM CNAME, WKD, MTA-STS,
etc.) are shown with green/red indicators but are **advisory** — the domain is activated
on challenge + MX only. WKD, MTA-STS, and DKIM become functional as their respective
DNS records resolve, which the background drift checker (ADR-0038) tracks.

**Rationale for challenge + MX as the activation gate:** MX presence confirms the user
has control of incoming mail routing, not just DNS read access. Without MX verification,
an attacker could add any domain they can edit DNS for, even if they cannot actually
receive mail on it.

### Polling endpoint (HTML fragment)

The domain detail page includes a per-record status panel. `partials.js` polls
`GET /api/v1/domains/{id}/verification-status` every 5 seconds while the page is open
and the domain is in `pending` or `partial` state. The endpoint returns an **HTML
fragment** (not JSON), consistent with ADR-0012.

The fragment shows:
- Overall status chip (pending / partial / verified / drifted).
- Per-record row: record name, type, expected value, actual value (truncated if long), 
  ✓ or ✗ indicator.
- "Refresh now" link (stops the polling loop and triggers an immediate re-check).

Polling stops when the fragment includes a `data-poll-stop="true"` attribute (status
reached `verified`). `partials.js`'s `poll()` primitive already handles this pattern.

### DNS lookups

Implemented with Go stdlib `net.LookupTXT`, `net.LookupMX`, `net.LookupCNAME` — no
external DNS library. Lookups run with a 10-second context timeout per record. NXDOMAIN
is treated as "record missing" (not an error). Temporary DNS failures (SERVFAIL,
timeout) are treated as "unknown" — the per-record indicator shows a grey "?" rather than
red "✗", and the overall verification does not fail.

### Token revocation

When a domain is deleted from the instance (by the user or via `psql`), the
`verification_token` is invalidated (domain row is deleted). If the user re-registers the
same domain, a fresh token is generated.

## Consequences

**Positive:**
- DNS-only verification requires no HTTP redirect infrastructure and works before the
  domain points to this instance.
- The 7-day token expiry is long enough to accommodate slow DNS propagation (including
  weekend registrar support queues).
- Partial-progress: the user can publish records in multiple sessions; the wizard remembers
  the token and current state.
- HTML-fragment polling keeps the UI live without a full-page reload or a custom WebSocket.

**Negative / trade-offs:**
- The `_rookery-challenge` TXT record must be removed by the user after verification —
  it is no longer needed and is harmless if left, but it is mild DNS clutter. The UI
  notes this after verification succeeds.
- Two-record activation gate (challenge + MX) means users who only publish the challenge
  TXT do not get a verified domain. The UI explains this and the MX record is shown first.
- The DMARC `rua` address in the pre-filled record uses the user's domain address
  (`dmarc@alice.com`) before that address has been activated — the user may need to
  create that address manually in their DNS mailbox or configure it after activation.

## Alternatives considered

- **HTTP redirect verification (user proves control by returning a file at a well-known
  path):** rejected. Requires the domain to already point at the instance, which is the
  thing we're trying to verify.
- **Email verification (server sends a code to postmaster@<domain>):** rejected. Bootstrap
  paradox: `postmaster@<domain>` doesn't exist on the instance until the domain is verified.
  Could route to an external address, but that assumes the user has email access at the
  domain before the instance is set up — exactly the scenario we're trying to avoid.
- **Immediate activation on challenge TXT only (no MX check):** rejected. An attacker
  who can add TXT records to a domain they don't operate could register it and intercept
  mail if MX is also pointed here. Challenge + MX together confirms both DNS control and
  routing control.
- **Background auto-polling every N seconds server-side (not user-driven):** considered.
  A background worker could continuously check all pending domains. Rejected for Phase 4
  because user-driven polling is sufficient and simpler; a background worker that fires a
  DNS query for every pending domain every 30 seconds is wasteful. ADR-0038's drift
  detector covers re-checking verified domains on a scheduled basis.
