# ADR-0009 — Desktop-first, mobile-tolerated

**Status:** Accepted  
**Date:** 2026-05-15  
**Deciders:** project founders

---

## Context

The target user is a technologist who works primarily on a desktop or laptop: a real
keyboard, a file system, `gpg`, an `$EDITOR`, and the ability to store a long-lived PGP
private key on durable local storage they control (P9 of PLAN.md). This choice affects
every UI decision and must be stated explicitly so it is not eroded by well-intentioned
"can we make this work on mobile too?" PRs.

## Decision

**The web UI targets desktop browsers on large viewports.** This is the primary design
constraint, not an afterthought.

**Mobile browsers will load the same UI and basic flows will work.** We use a small
number of CSS breakpoints to prevent multi-column layouts from breaking on narrow
viewports. That is the entire mobile story in v1.

**We do not:**
- Optimize touch targets, gestures, or input for phones.
- Ship a native iOS or Android app.
- Implement PWA installability (no `manifest.json`, no service worker).
- Add push notifications (server-sent events while a tab is open; that's it).
- Optimize layouts for thumb-zone ergonomics.

**Smartphones are not good places to handle the recovery file.** This is a considered
position, not a lazy omission:
- Phones are lost, lent, sold, and replaced on 2–3 year cycles.
- Mobile file pickers make the recovery-file export flow awkward: the file is
  typically dumped into a generic "Downloads" folder with no clear hand-off to a
  password manager or secure-notes app, and users routinely back the entire Downloads
  folder up to cloud storage they may not control.
- rookery's key model (ADR-0003) requires the recovery file on *every* login — there
  is no IndexedDB or other persistent-browser cache to soften the experience. On a
  desktop this is a known cost the target user accepts (P9). On a phone it is a
  significantly worse experience that we are not going to optimize for in v1.
- A user who sets up rookery on their phone and loses the phone is in the same position
  as any other "lost device" scenario — they need the recovery file. But phones make
  the recovery-file step feel more like a failure and less like expected key hygiene.

**Future mobile path.** If there is sustained demand from the actual user base after
v1, and a contributor wants to own it: a PWA first, a native client only if the PWA
hits real limits. This requires a separate design conversation about key storage on
mobile. It is Phase 7+ at the earliest. Not a v1 gap.

## Consequences

**Positive:**
- The design is free to use full-page layouts, dense information display, form patterns
  that work well with physical keyboards, and file-system operations (export recovery
  file, drag in recovery file) that are friction-free on desktop.
- We do not ship two UIs with half the quality of one.
- The CSS can be simpler: fewer breakpoints, no touch-specific states, no mobile-first
  media query inversion.

**Negative / trade-offs:**
- Users who primarily access email from a phone will have a poor experience. This is
  deliberate; they are explicitly not the v1 target audience.
- The "must export recovery file" step in onboarding is significantly more awkward on
  mobile (where file saves are managed by the OS, not a visible file system). We
  document this as a reason to prefer desktop for setup.
- No push notifications means mobile users who happen to access the site on a phone
  will not get background new-mail alerts. Acceptable given the audience.

## Alternatives considered

- **Mobile-first design (build for mobile, scale up to desktop):** rejected. Mobile-
  first design optimizes for small touch targets and simple flows. Our target user does
  not need either of those things, and our use of file-system affordances (recovery
  file) is hostile to a mobile-first paradigm.
- **Separate mobile-optimized HTML + CSS served to narrow viewports:** rejected. This
  is effectively shipping two UIs. The breakpoint approach (stop multi-column layouts
  from breaking) is the correct minimum.
- **React Native or Flutter cross-platform app:** rejected. Not in scope for v1 under
  any framing. The project does not have the contributor capacity to maintain a mobile
  app in addition to the server and web client.
- **PWA in v1 with service worker for offline inbox access:** rejected. Service workers
  that cache decrypted mail are a significant security consideration; the PWA
  installability story on iOS (Safari) is still poor; and the target audience does not
  need offline access — they have a laptop.
