# ADR-0004 — Key discoverability: WKD + auto-attach on outbound; Autocrypt later

**Status:** Accepted  
**Date:** 2026-05-15  
**Deciders:** project founders

---

## Context

For PGP encryption to be the default between consenting parties, each sender must be
able to find the recipient's public key without manual intervention. rookery must both
*publish* its users' public keys in a discoverable way and *discover* recipients' keys
before sending.

## Decision

**Publishing:** rookery publishes each user's public key via the **WKD advanced method**
only (see ADR-0024 for the method choice). The WKD endpoint is served at
`https://openpgpkey.<domain>/.well-known/openpgpkey/<domain>/hu/<z-base-32-sha1>`.
No WKD direct method.

**Inbound auto-attach:** rookery auto-attaches each user's public key to every outbound
message as a `application/pgp-keys` MIME part. Receiving mail clients (Thunderbird,
Proton, other rookery instances, any RFC 3156-conforming client) can harvest the key
for future encrypted replies without performing a WKD lookup.

**Outbound discovery:** before composing, the server attempts to discover the
recipient's public key by checking, in order:
1. The per-user known-keys cache (keys harvested from inbound auto-attaches or from
   prior WKD lookups).
2. WKD lookup at the recipient's domain.
3. Optionally, a configurable public keyserver (e.g. `keys.openpgp.org`). Off by
   default; operator opt-in.

Results are shown to the user in real time on the compose page via `partials.js`.

**Autocrypt:** not in v1. The Autocrypt specification (header-based key gossip) is a
Phase 7+ addition. It is architecturally compatible with this design: Autocrypt headers
are parsed from inbound mail and can feed the known-keys cache. We defer it because WKD
+ auto-attach cover the main cases for the v1 audience.

**Trust model:** all key discovery is TOFU (Trust On First Use). When a key is first
associated with an address, it is stored with a "first seen on <date>" indicator. Users
can compare fingerprints out of band. Fingerprint verification UI is v1; key
transparency logs (out of scope) and Autocrypt history (Phase 7) are long-term
mitigations for the TOFU problem. §9 of PLAN.md discusses the key-discovery TOFU
problem honestly.

## Consequences

**Positive:**
- WKD is supported by Thunderbird, GPG, Proton, Fastmail, and other major clients —
  rookery users' keys are discoverable by the ecosystem without any special setup.
- Auto-attach means correspondents receive the key in-band with the first message;
  WKD is a backup for prospective senders who compose before receiving.
- The known-keys cache means subsequent messages in a thread are encrypted
  transparently, without repeated lookups.

**Negative / trade-offs:**
- TOFU is vulnerable to MITM at first contact (a malicious server or DNS attacker can
  serve a wrong key for a new correspondent). This is documented as the "hardest
  unsolved issue" in §9 of PLAN.md. The mitigations (fingerprint UI, optional
  keyserver cross-check, future Autocrypt history) reduce but do not eliminate the
  risk.
- WKD advanced method requires the instance to control `openpgpkey.<domain>` TLS —
  this is part of the per-domain ACME provisioning in Phase 3.
- The z-base-32 SHA-1 hashing of local-parts for WKD paths is non-trivial to implement
  correctly; interop testing is explicitly budgeted in Phase 1.

## Alternatives considered

- **WKD direct method as well:** rejected in ADR-0024. The direct method requires
  serving WKD paths from the domain apex, which conflicts with users who run their own
  website there.
- **DANE / OPENPGPKEY DNS records:** interesting, but requires DNSSEC, which most
  registrars and users do not have. Deferred to Phase 7+ if the ecosystem matures.
- **Keyserver-only discovery:** less reliable and more centralized than WKD. Keyservers
  are a complement to WKD, not a replacement.
- **No auto-attach:** Thunderbird users have come to expect auto-attach; it is the
  primary way keys get into the ecosystem for users who send before receiving. Rejecting
  it would hurt interop.
