# DNS reference

Every DNS record that rookery needs, what it does, and what happens if it's missing.

Use `./rookery check-dns` to see the current propagation status for all records.

## Records for the primary domain

Replace `rookery.example` with your actual domain throughout.

### A / AAAA — address records

| Name | Type | Value |
|---|---|---|
| `rookery.example` | A | your server's IPv4 |
| `rookery.example` | AAAA | your server's IPv6 (if available) |
| `openpgpkey.rookery.example` | A | same as above |
| `openpgpkey.rookery.example` | AAAA | same as above |
| `mta-sts.rookery.example` | A | same as above |
| `mta-sts.rookery.example` | AAAA | same as above |

**Why:** Caddy needs address records for `openpgpkey.*` and `mta-sts.*` to
provision Let's Encrypt certificates via HTTP-01. Missing these means WKD key
publishing and MTA-STS won't have TLS — WKD requires HTTPS per spec.

### PTR — reverse DNS

Set the PTR record for your server's IP at your VPS provider's control panel.
It should resolve to your primary domain or a subdomain (e.g. `mail.rookery.example`).

```
1.2.3.4 → rookery.example   (or mail.rookery.example)
```

**Why:** Many receiving mail servers check that the connecting server's IP has a
PTR record and that the PTR matches a forward DNS lookup. A missing PTR will cause
your outbound mail to be rejected or flagged by many destinations. **Set this
before anything else** — propagation can take 24–48 hours.

### MX — mail exchange

| Name | Type | Priority | Value |
|---|---|---|---|
| `rookery.example` | MX | 10 | `rookery.example.` |

**Why:** Tells the world where to deliver mail addressed to `*@rookery.example`.
The MX target must be the bare primary domain (no `mail.` prefix) because rookery
listens on port 25 on the same host that serves the web UI.

### SPF — sender policy

| Name | Type | Value |
|---|---|---|
| `rookery.example` | TXT | `v=spf1 a -all` |

**Why:** Authorises your server's IP to send mail from `@rookery.example`. `-all`
rejects mail claiming to be from your domain that didn't come from your server.
If you're using a smarthost relay (Phase 8), change `a` to `include:_spf.relay.example`.

### DKIM — signing keys

rookery generates two DKIM keypairs on first run: ed25519 (primary) and RSA-2048
(fallback). The public key values are logged at startup. Read them with:

```sh
./rookery logs | grep -i dkim
# or
./rookery check-dns
```

| Name | Type | Value |
|---|---|---|
| `rookery-ed25519._domainkey.rookery.example` | TXT | `v=DKIM1; k=ed25519; p=<base64>` |
| `rookery-rsa._domainkey.rookery.example` | TXT | `v=DKIM1; k=rsa; p=<base64>` |

**Why:** DKIM proves outbound mail is signed by your server. Without it, receiving
servers can't verify your identity and DMARC policy won't work. ed25519 is the
primary (smaller, faster); RSA-2048 is the fallback for receivers that don't
support ed25519 (still common in older infrastructure).

### DMARC — authentication policy

| Name | Type | Value |
|---|---|---|
| `_dmarc.rookery.example` | TXT | `v=DMARC1; p=quarantine; rua=mailto:postmaster@rookery.example` |

**Why:** DMARC tells receiving mail servers what to do with mail from your domain
that fails SPF or DKIM. `p=quarantine` means "put in spam folder" (not `reject`,
which would drop it silently). The `rua` address receives aggregate reports from
major providers — useful for diagnosing alignment problems.

Start with `p=quarantine` (not `p=reject`) until you've confirmed your DKIM and
SPF are working correctly via the aggregate reports.

### MTA-STS — strict transport security

| Name | Type | Value |
|---|---|---|
| `_mta-sts.rookery.example` | TXT | `v=STSv1; id=<server-generated>` |

The `id` value is generated by the server. Read it from `./rookery check-dns`.

rookery also serves the MTA-STS policy file over HTTPS at
`https://mta-sts.rookery.example/.well-known/mta-sts.txt`. Caddy provisions
the certificate for `mta-sts.rookery.example` automatically.

**Why:** MTA-STS tells well-behaved sending mail servers that they must use TLS
when delivering to your domain, and what certificate to expect. Without it, your
inbound mail can be intercepted in transit by a network attacker who can strip
TLS. Missing this record is not immediately breaking but is a deliverability and
security gap.

### TLS-RPT — transport failure reporting (recommended)

| Name | Type | Value |
|---|---|---|
| `_smtp._tls.rookery.example` | TXT | `v=TLSRPTv1; rua=mailto:postmaster@rookery.example` |

**Why:** TLS-RPT causes major sending mail servers to send you reports about TLS
failures when delivering to your domain. Useful for diagnosing certificate
problems before they become delivery outages. Not required, but strongly
recommended.

## Records for custom user domains

When a user brings their own domain (e.g. `alice.com`), the web UI shows them the
full set of records to publish. The set is the same as above, but all DKIM CNAMEs
point to the primary domain's keys:

| Name | Type | Value |
|---|---|---|
| `rookery-ed25519._domainkey.alice.com` | CNAME | `rookery-ed25519._domainkey.rookery.example` |
| `rookery-rsa._domainkey.alice.com` | CNAME | `rookery-rsa._domainkey.rookery.example` |
| `openpgpkey.alice.com` | CNAME | `openpgpkey.rookery.example` |
| `mta-sts.alice.com` | CNAME | `mta-sts.rookery.example` |

The MX, SPF, DMARC, MTA-STS TXT, and TLS-RPT TXT records need to be published
by the user on their own domain. The web UI generates the exact values during
domain setup.

## Record propagation times

| Provider type | Typical TTL | Propagation |
|---|---|---|
| Cloudflare, Namecheap | 300s (5 min) | 5–15 minutes |
| Most registrars | 3600s (1 hour) | 1–4 hours |
| Some legacy registrars | 86400s (24 hours) | up to 24 hours |

`./rookery check-dns --resolver 9.9.9.9` bypasses your local DNS cache to check
Quad9's view of propagation.
